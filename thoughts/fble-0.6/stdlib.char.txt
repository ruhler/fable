Char@
=====

This will be a fun one.

It makes sense to support unicode characters in the standard library. So Char@
represents a unicode character.

Expected operations:
* Construct a character from a literal or unicode code point.
* Check for character equality, or ordering based on unicode code point.
* Get the unicode code point for the character.
* Standard character type classification: isspace, isupper, etc.
* Perhaps standard toupper, tolower kinds of transformations.

Existing fble API:
* Char@, with most of printable latin chars, and 'unknown' byte.
* Chars collection of literals.
* Standard interfaces: Eq, Show, etc.
* Conversion to/from ASCII integer value.
* Character types: IsUpper, IsLower, IsAlpha, IsDigit, IsAlNum, IsSpace,
  IsNewline
* Char@ can currently be used as a literal, implicitly.

I'll want to generalize to support all of unicode presumably.

C99:
* Character literal: source file encoded character literal.
  "mapped in an implementation defined manner to the execution character set".
* Escape sequences for ', ", ?, \, octal, hex
* Escape sequences for \a, \b, \f, \n, \r, \t, \v
* Conversions between char* and wchar_t (mbtowc)
* ctype.h:
  - behavior depends on current locale
  - isalnum, isalpha, isblank, iscntrl, isdigit, isgraph, islower, isprint,
    ispunct, isspace, isupper, isxdigit,
  - tolower, toupper
  - corresponding wide character versions of the above.

Unicode:
* Characters have code points.
* Major category: Letter/Mark/Number/Punctuation/Symbol/Separator/Other
* Minor category: uppercase/lowercase/titlecase/modifier/other
                  spacing/nonspacing/combining/enclosing
                  decimal/digit
                  connector
  etc.
* Basic type: Graphic
* Character: Character
* It's pretty complicated in general. I'm not sure standard library for
  Char@ is the right place to deal with these complexities a Unicode specific
  library would be better suited. Just pick the basics for Char@.

* Spaces: newline, space, tab, CR
* Symbols: !"#$%&'()*+,-./:;<=>?@[\]^_`{|}~

* Letters: A-Z, a-z

Differences between my characters and C99:
* C99 also has: \0, vertical tab, form feed
* C99 also has: alert, backspace
* C99 doesn't have '$' or '@' or '`'.

What to include in my character definition:
* The Ascii@ type, which is one big enum type.
* Conversion from Ascii@ to unicode code point for Char@.
* An Ascii char literal function that takes List@<Ascii@> to Char@.
 - We could include support for parsing unicode escape sequences and other
   escape sequences here if we wanted to.
* Optional: conversion from Char@ to Ascii@, for use in case expressions
  rather than force use of Eq?
* Implementations of character class functions: IsSpace, IsUpper, etc.?

What not to include in my character definition:
* Eq, Ord, Show. Anything that would tempt people to use it for anything other
  than character literals.

Limiting to ascii is good, because then if somebody wants to, they could write
their program entirely in ascii. No unicode support required by their
implementation. That is, use of funny characters beyond ascii in standard
library can be restricted to specific modules you can easily avoid including
if you want.

But we still have a composability challenge. If only there was an easy way to
say, given Ascii@ and Latin1@, form a composite string literal that
supports characters from both ranges. If only we had that, then everything
would be real easy:
* Define separate modules for separate unicode code pages, lazily.
* Have a way for a user to define a string literal given a list of types from
  various code pages. For example, something like Str = StrFor[Ascii@,
  Latin1@], etc.
