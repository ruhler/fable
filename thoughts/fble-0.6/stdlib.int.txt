Int is the first somewhat complex library to provide. It's worth taking some
time to plan it out in more detail. And compare more with other, existing
libraries.

High level: I want an unbounded, signed integer type. Don't include bitwise
operations.

Minimum needed to work with ints in practice:
* Int@ type, abstract.
* Some way to construct any Int@ value:
 - 0, Neg, Add1, Mul2
* Some way to deconstruct any Int@ value:
 - IsZero, IsNegative, IsPositive, Div2

What are the questions / decisions to make here?
* Int@ type is clear.
* Literals: how many to define in the language?

I need to review integer division again to understand what the options are and
what choice I'm using.

Options (from wikipedia 'Modulo'):
* Euclidean:  a = b*q + r, 0 <= r < |b|
  Common mathematical definition, number theory.
* Truncated:  a = b*q +r , q = trunc(a/b)
  Remainder has same sign as 'q'.
* Floor:      a = b*q +r , q = floor(a/b)
  Remainder has same sign as 'b'.
* Rounded:    a = b*q +r , q = round(a/b)
  IEEE 754
* Ceiling:    a = b*q +r , q = ceil(a/b)

What was my argument before?

My argument there was fairly convincing. Use 'floor' so that sign of remainder
has the same sign as the divisor. I think if modulo n as giving a number
between 0 and n, so that makes sense. It avoids 'singularity' like behavior
around 0.

   5 =  2 * 2 + 1
  -5 =  2 * -3 + 1
  -5 = -2 * 2 - 1
   5 = -2 * -3 - 1

Euclidean would instead say:
  -5 = -2 * 3 + 1
   5 = -2 * -2 + 1

So, interesting cases are:
   5 /  2 = 2 remainder 1
  -5 / -2 = 2 remainder -1 or 3 remainder 1
   5 / -2 = 3 remainder -1 or 2 remainder 1
  -5 /  2 = -3 remainder 1

Honestly, I rather have that a / -b = -b / a, and that 
a % -b != -a % b. The modulo makes sense to be different to me, the division
does not. Floor gives us that. So I vote we go with floor. Good.

Abstractly we have choices of either hard coding radix or having radix encoded
as part of the syntax. For example:

* ReadDec, ReadHex, ReadOct, ReadRadix(n), etc... which all take a specific
  radix.
* ReadInt, which can take, e.g. 0x123, or 0013, or 123, and so on.

So decisions are:
* Which radix specific functions to support?
 - At least Bin, Oct, Dec, Hex corresponding to Int literals?
* For a give radix, what syntax to support?
 - '+', '-', leading zeros allowed? '_' or other separators? ',' separator?
   whitespace?
* Should we provide a mixed radix read function?
 - If so, what syntax and what radix to support within that?

Let's see what other libraries use.

C:
* atoi: base 10
 - any amount of whitespace, then single optional '+' or '-'
* strtol: base as argument, or special 0.
 - If 0, then prefix of '0x', or '0', or else. Except 16 accepts 0x too.
 - Reads to the first invalid character.
 - But other things can be accepted depending on locale if not C or POSIX.

Java:
* Integer.parseInt
 - Optional explicit radix to pass.
 - Optional single '+' or '-' followed by all valid digits for the base.
 - Throws exception if malformed.

Go:
* ParseInt
 - Like strtol. Prefixes for bases are 0b, 0 or 0o, 0x
 - Underscores allowed if radix is special '0' only.

Python:
* int()
 - Can have whitespace around it, leading zeroes, underscore separators.
 - Radix prefixes are 0b/0B, 0o/0O, 0x/0X

Proposal:
* /Std/Int/Parse%
  - (Int@, String@) { Maybe@<Int@>; } Radix
    Parse an integer in the given radix, radix between 2 and 36.
    Returns Nothing for bad radix or failure to parse.
    Syntax is optional '+' or '-', followed by the radix digits.
    Any number of leading zeros (after '+' or '-') are allowed.
    Single '_' separators are allowed.
  - (String@) { Maybe@<Int@>; }; Int
    Parse an integer, where radix is part of the syntax.
    Syntax is optional '+' or '-'.
    Radix is '0b' '0B' '0o' '0O' '0d' '0D' '0h', '0H', '0x' '0X'
      Anything else following an initial '0' is not allowed.
    If the first digit is [1-9], then radix is 10.
    After the radix specification, any number of leading zeros are allowed.
    Single '_' separators are allowed.
* /Std/Int/Lit%
  - Bin, Oct, Dec, Hex - wrappers around Radix(2), Radix(8), ...
    Taking List@<Ascii@> as arguments.
    Assuming no error.
  - Int - wrapper around /Std/Int/Parse%.Int.

I have a bootstrapping problem:
* Parse.Radix depends on character literals to compare against 'a', etc.
* Character literals depend on integer literals to define 'a' as Int|97.
* Integer literals depend on radix to parse the 97 into an integer.

It feels like the right way to break the cycle is for characters to continue
to depend on integer literals, but not have integer literals depend on strings
or characters. Because Char@ depends on Int@, not the other way around.

The only way I can do that without duplicating the parsing logic would be to
implement the parser on a special letter type that Int knows about. It would
bootstrap on something that takes a list of Int@ for composition rather than
a list of letters. The string version of the parser would then map string to
list of int literal letters and call the int literal letter parser.

Why not reuse Ascii@ for int literal letters? As long as I can separate the
definition of Ascii@ from any use of int literals, it has what I want. I'm
defining it anyway. Then everyone can use Ascii for int literals.

I suppose that's another consideration. Today I have IntLit@, BinLit@, etc.
for literal types. If all those types were the same, I could write a function
that works for generic literals more easily. Otherwise I have to have multiple
variations. It seems like we should use one letter type to cover all the kinds
of literals. Though that means you can have invalid characters in literals.

Reviewing my current implementation, I suppose all we really need is a map
from letter to Int@ and a radix. Then some sort of special handling for '+',
'-', separators, and radix prefix. But that kind of handling can be done
separately anyway, because it all comes at the front. Well, except for the
separators.

Partly the question is, if I've already defined integer literals without use
of Char@ or String@, is there any justification for defining an integer
parsing function as part of the standard library? Because almost certainly
it's not going to use the same syntax as whatever you application requires.

Maybe it's enough for it to be a superset of whatever syntax your application
requires? Like, if it can parse +/- sequence of digits, without worrying about
separators or radix specifications, that's enough? Is it fair not to worry
about radix specifications or leading zeroes?

Or maybe we want to have some specification you have to supply for your number
format? Like how they do for dates? It seems pretty clear what all choices
there are, doesn't it? Maybe not. But if the tedious part is mapping digits to
ints and combining those together by radix, then all we ought to need is the
pure parsing that does that. You can implement your own wrapper as needed. For
example:
* Check for and strip of +-.
* Check for and strip of 0x, 0, 0o, 0b, etc to pick radix.
* Remove leading whitespaces. Remove internal separators.
* Then call the shared function that knows how to convert to int and compose
  together.

* To parse DIMACs integer.
* To get n on command line for benchmark games

All these cases really want something simple and basic I think. For the most

separators. Decimal only. Like, basic atoi stuff.

Why don't we make that our spec? Optionally we could have radix specified
externally, but otherwise, a straight forward:

 ParseInt(radix, string)

Where radix is between 2 and 36. String is optional +, -, followed by the raw
digits. No separators. No 0x h whatever. No reason to have to do anything or
share anything at all with integer literals.
