Show@
=====

Show@ class is wanted for unit testing: be able to print out the value of a
data type in human readable format, so you can tell in a test error message
what went wrong.

Typical use is for testing and debugging. You could see it used for
interacting with humans, for example in a tictactoe game.

The simplest approach is just (A@) { String@; }. The concern with that
approach is performance. If you have a bunch of Show instances implemented as
concatenation of show instances, this leads to lots of intermediate strings
and copies. If, on the other hand, you can write it in such a way that it
prepends to an existing string, you can avoid all those intermediate strings
and copies. It is significant from a complexity point of view.

My goals:
* Implementation of Show should feel natural.
* Avoid quadratic blowup for showing large composite objects.

How do we use the result of Show? Maybe we don't need to produce a string at
all. What if we are going directly to an output stream?

New idea: iterators.

I claim we can return exactly what we want for show as a data structure which
looks very much like an iterator:

  I@ = M@< +(*(T@ next, I@ tail) items, Unit@ empty) >;

You invoke monadic sequencing to access the next item (or end).

For example, say M@<A@> = (Unit@) { A@; }, and T@ is String@:

  I@ = (Unit@) { +(*(String@ next, I@ tail) items, Unit@ empty); };

Show = (List@ l, Unit@ _) {
  l.empty.?(R@(empty: Unit));
  R@(items: @(next: ...

Anyway, you can imagine having a library function to take two iterators and
produce one that goes through sequentially:

  (I@, I@) { I@; } Seq = (I@ a, I@ b) {
    (Unit@ _) {
      R@ r = a(Unit);
      r.?(empty: b(Unit));
      r.@(tail: Seq(tail, b));
  };

Show = (List@ l, Unit@ _) {
  l.empty.?(R@(empty: Unit));
  Seqs[Show(l.items.head), ShowStr|':', Show(l.items.tail)];
};

Notice that the recursive applications to Show are partial, so they will
capture the input argument without executing the body of the show. That's how
you avoid looking at the end of the list until you've already done something
useful with the front.

We get different things depending on choice of M@ here:
* Pure@ - A straight up list of T@.
* (Unit@) { A@; } - A lazy list of T@, without intervening side effects.
* M@<A@> - A generic list of T@, which can be interleaved with side effects.

The general monadic sequencing would look like:

  (I@, I@) { I@; } Seq = (I@ a, I@ b) {
    R@ r <- m.do(a);
    r.?(empty: b);
    m.return(r.@(tail: Seq(r.tail, b)));
  };

For pure, that would turn into?

   a.?(empty: b);
   a.@(tail: Seq(a.tail, b));

Which is exactly what append looks like. It's your quadratic blowup for
string@. Especially of T@ was Char@ instead of String@. Then we have literally
the String@ type.
