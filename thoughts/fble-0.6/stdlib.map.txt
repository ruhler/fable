Map
===
We want the standard library to include a map data structure that efficiently
maps from keys to values.

I suppose you could use a pure fble function for this, but that's not
efficient if you want to do inserts and removals without ending up with a
linear search for lookup.

You could also use a list of key value pairs if you're okay with linear
search.

I have an implementation of a balanced binary tree map which should give more
efficient lookups.

And you can imagine fancier structures based on the structure of key types or
hash map like things.

What I need to decide is:
* The API for interacting with the map.
* The approach for defining the map:
 - Use balanced binary tree map? Or a small number of predefined map types
   (e.g. std::map, std::unordered_map)
 - Or let people define their own maps based on the key type.

How Map looks for me today:
* Map@, Key@, Empty, IsEmpty, Insert, InsertWith, Delete, Lookup, ForEach
* We have custom map implementations per key type:
  Char@, IntP@, Int@, List@, String@
* We have a generic balanced tree map implementation.
* We have a Set@ wrapper around Map@.

The custom map implementations are essentially hard coded generic data type
based implementations.

Specifically:
 Map@<Unit@> is implemented as Maybe@<Value@>
 Map@<+(A@, B@)> is *(Map@<A@>, Map@<B@>)
 Map@<*(A@, B@)> is Map@<A@, Map@<B@>> (in theory, do I actually have it?)

In practice it's tedious enough to implement a map that I use my generic Cmp@
based tree map for anything that isn't a performance concern.

I should think I could provide helper functions to define these generic maps
as compositions of things. It's a question of whether the performance
overheads of that kind of composition are worth the performance improvements
of not using a generic tree map.

From an API point of view, the ideal thing would be to have a single
implementation of Map@ that works for any key and value type and that is fast
enough for what you want.

Consider an Int@ type though. If we use a tree map, we are doing comparisons
between int values. The cost to traverse the entire Int@ value for a single
comparison is probably comparable to the cost of traversing the map to find
the associated value. That makes me think tree map is always going to be
slower and algorithmically more complex than a custom map implementation.

The implementation of Char@ map I have today is instructive, because we are
implementing a Char@ map in terms of an Int@ map. What we need for that to
work is a way to convert the Char@ key to an Int@ key and back. We need a way
to go back for ForEach.

For a shallow conversion from Char@ to Int@, that's probably not any concern.
But for something like Int@ or List@ converting to the equivalent generic type
recursively? Maybe it's not so bad actually? If we define our maps
recursively?

I suppose what I'm getting at is, maybe I can start by rewriting my existing
generic map implementations using generic map composition functions. See how
that feels. Based on the idea that union becomes a pair, pair becomes a nested
map.

If I'm going to do that, why make this map specific? Why not have a generic
'Data' type that you can convert any generic data type like things into, and
then provide an implementation of map that works on anything you can convert
into Data?

If we want to allow for shallow wrapping and unwrapping of data, maybe Data@
looks something like:

 Data@ = ...?

I'm not sure how it would work. Full unrolling would be something like:

 Data@ = +(Unit@ unit, +(Data@ a, Data@ b) union, *(Data@ a, Data@ b) struct);

I guess shallow would be something like:

 Data@ = <@ A@, @ B@, @ C@, @ D@> { +(
   Unit@ unit, +(A@ a, B@ b) union, *(C@ a, D@ b) struct
 ); };

Or skip declaration of a Data@ type, and have direct helpers:

 UnionMap is a function that takes as input:

   Map@<A@>, Map@<B@>, (T@ -> +(A@, B@)), +(A@, B@) -> T@)

 And returns a map of type Map@<T@>.

Should Map@ be a builtin type? Think of it like an array?

Depends how you want to define it. I suppose you provide a default value?
Inserting into the map will be expensive, but we could make lookups fast?

Or, wouldn't it be nice if we had a way to write a function that generates
code for us based on the structure of a type? I've wanted that before for
other use cases too. Essentially what you could do with a Data@ class, but
hopefully magically without the overhead cost of converting to/from Data@.

---

I don't want users to have to write their own map implementation for custom
data types. Or at least, they shouldn't have to know the general approach of
union to struct, struct to map of map.

Options are:
* I provide helper functions to make it easy to define their own map.
* I provide a hash map implementation of Map, if you provide a way to hash and
  compare your types.
* I provide a 'data' implementation of Map, if you provide a way to convert
  your types to data.
* Optionally you could provide a way to go back to your type from data, then
  we don't have to save the key on the map, we can have it encoded
  implicitly in the structure of the map. Without sharing though.

I haven't thought of a reasonable way to describe generic code in fble yet
that could let you write a generic implementation of customized Map (or Eq, or
Data, or Has) that works for any use provided data type.

We could probably do a lot of sketchy but powerful stuff if we had a standard
way of serializing a data type. For example, say we provide a way to represent
any data type is a sequence of Int@. When you define a data type, you provide
functions to/from a sequence of Int@. We can make the API a lazy one, so we
don't have to traverse the entire object to get the initial part of the
sequence.

That sounds so dangerous though. Like it subverts type checking. Also, tedious
because the runtime knows all the information it needs to do the conversion
automatically. It's annoying the user would have to write all that code. I
really wish we could write code that operates on the native representation of
the data type, and be able to reuse the same code for any purely data data
type.

I'm wondering whether Hash@ should be lazy or not. So we could do partial
lookup in a map, find that the key can't be there without having to hash the
entire key.

It occurs to me that lazy Hash@ would be very similar to Output@<Int@>. Maybe
the Hash class should be:

  Hash@ = <@ A@>(A@) { Output@<Int@>; };

You can hash your value to a sequence of ints. Output@ lets you describe the
hash lazily.

I should be able to produce this pretty easily. Something like:

 (List@<A@> x) {
   x.?(nil: Put(Int|0));
   Unit@ _ := Do(Put(Int|1));
   Unit@ _ := Do(hash(x.items.head));
   Do(hash(x.items.tail));
 };

Hmm... does that give us enough structure to go back the other direction? How
do we know when one field of a struct ends and the next one starts? Or it
doesn't matter, we always go in order?

Anyway, assuming that's not an issue, could we consume this for map?

It would be like having a map where the key is a list of Int@, but we pass the
key as an Output@<Int@> that it can consume a little at a time? Could we
implement the map as the stream type S@?

 Map@ = +(Unit@ empty, Maybe@<V@> value, IntMap@<Map@> s);

 S@ = (Map@) { Map@; }

 return = (Map@ m) { m; };
 do = (S@ a, (Unit@) { S@; } b, Map@ m) {
   m.?(empty: m);   # Doesn't matter what you think you have.
   Map@ n = a(m);   # Look up based on a key information.
   n.?(empty: n);   # Doesn't matter what you have in b.
   b(Unit, n);      # Look up based on rest.
 };

 put = (Int@ v, Map@ m) {
   m.s.get(v);
 };


Then, at the top level, we simply run your Output@ computation on our initial
Map@. At the end, we return the value if found, or nothing.

It may be we need to use a union of struct instead of a union of union. But I
think the rest stays the same.

Thus we get efficient lookup from your value in a lazy way. It's easy to
implement Hash@.

Other questions:
* How hard would it be to use this Hash@ implementation to implement Eq@?
  Cmp@?
* How hard would it be to implement a reverse hash implementation, assuming
  objects are perfectly hashed?

The reverse hash, I fear, needs more of a monadic interface? Or some way to
lazily produce the sequence of Int@. But I'm sure I can do that. It's like
taking an Output@ and returning back an In@ that I can use to consume the
state of the Output@. So reverse interface is 

  RHash@ = <@ A@, <@> M@>(Monad@<M@>, M@<Int@>) { A@; }

And we implement it something like:

  RHashList = (Monad@<M@> m, M@<Int@> in) {
    Int@ x := m.do(in.get);
    IsZero(x).?(true: m.return(Empty<A@>));
    A@ head := m.do(RHashA(m, in));
    List@<A@> tail := m.do(RHashList(m, in));
    m.return(Items(head, tail));
  };

Easy, no?

And if you think of hashing like this, it's also like a way to serialize data.
Like, I could write out to a file and read back in.

So, we use Output@, In@ stream based interface for hashing and unhashing. If I
have names for Serialize@ and Deserialize@, then we could provide two
different map implementations. One that takes Serialize@ and Eq@, assuming two
keys could serialize to the same value but not be equal. And a map that takes
Serialize@ and Deserialize@, assuming no two values serialize to the same
value. In this case, Deserialize@ is used to save space in the map... but
honestly, it just seems easier to store the key there and always use the hash
map approach, so you just need to pass Serialize@ and promise that defines
your notion of equality.

It's interesting that for the case of Serialize@ and Deserialize@... do I want
to make use of 'end' feature of In@ and Out@? Perhaps. It could save me from
storing a terminating 0 I suppose? Or maybe not. Maybe it doesn't compose well
at all.

There's an opportunity here to learn about performance and to improve the
Stream@ interface. Maybe. Or find a better interface for Data@ at least.

Let's see if we can find a balance between theory and practice that gives
something with decent complexity and good runtime performance.

Let's start with theory while waiting for perf to get installed.

Today:
A. Stream is: Forall S, Return, Do, Put: S
   Do is (S, (Unit@) { S; }) S;

This means Do function gets to decide when the next part of output is
traversed.

Some alternatives we could consider:
B. Stream is Forall S, Return, DoPut: S
   Like the List ForAll function. DoPut is (A@, S@) { S@; }
   DoPut = {
     Do(Put(a), (Unit@) { s; });
   };

C. Stream is Forall S, Return, Do, Put: S
   With Do (S@, S@) { S@; }
   Do'(a, b) = Do(a, (Unit@) { b; });
 
Practically speaking, (A) forces a function allocation for the Lazy@<S@>.

Let's take implementation of Stream for a generic binary tree as an example?
Or, how about a list.

A.
  (List@) { Stream@; }
  Puts = (List@ values)<@ S@>(Channel@<S@> ch) {
    values.?(empty: ch.return);
    Unit@ _ := ch.do(ch.put(values.items.head));
    Puts(values.items.tail, ch);
  };

B.
  (List@) { Stream@; }
  Puts = (List@ values)<@ S@>(Channel@<S@> ch) {
    values.?(empty: ch.return);
    ch.doput(values.items.head, Puts(values.items.tail, ch));
  };

C.
  (List@) { Stream@; }
  Puts = (List@ values)<@ S@>(Channel@<S@> ch) {
    values.?(empty: ch.return);
    ch.do(ch.put(values.items.head), Puts(values.items.tail, ch));
  };

Regardless of what you can come up with for the implementation of 'Do', (B)
and (C) force you to traverse the entire list before calling do.

Maybe this isn't a problem, because in all my current use cases for stream, I
want to traverse the entire thing anyway.

In either case, I can reverse the order of the traversal by allocating
functions. Something like:

  Do(a, b) = a(b(x));
  Do(a, b) = b(a(x));
  
It feels like the important question isn't traversal order then. It's whether
we are forced to traverse the entire thing first. Maybe also whether we can
pause and later resume from where we left off.

Abstractly, if I have a stream of elements I want to traverse, the kinds of
things I want to do are:
* Have control over when to access the next element in the stream.
* Have the ability to not access the next element in the stream.
* Have the ability to capture and return the rest of the stream without
  accessing it.

Perhaps the most direct way to do that would be if the stream was a lazy list.
Or had an API equivalent to that.
