Function Application Sugar
==========================

Motivation: I want a nice way to annotate a test as an expected failure.

Let's say I have a test case:

  Test(Str|'MyTest', (Unit@ _) {
    Foo@ wnt = ...;
    Foo@ got = ...;
    AssertEquals(eq, show, wnt, got);
  }

Now it starts failing, and I don't want to fix it. It would be great if I
could add a single line to the start of the test:

  Test(Str|'MyTest', (Unit@ _) {
    ExpectFails(Str|'!=') $
    Foo@ wnt = ...;
    Foo@ got = ...;
    AssertEquals(eq, show, wnt, got);
  }

That line is a function that takes a value. The following statement is the
value to provide to that function.

We've seen places where this kind of way to avoid indentation as a syntactic
sugar is really useful:
* fbld @@ syntax
* case statement syntax
* monadic bind syntax

How about something for function application too?

We can even think about generalizing bind syntax somehow. Maybe figure out a
way to make it work with no-argument bind?

Straw proposal to get thoughts going:

  <expr> $ <stmt>

Is desugared to: <expr>(<stmt>)

Now here's where things get a little tricky. Do we evaluate the 'body' before
calling the function? As proposed above, yes. Works great for ExpectFails.
Does not work great for a 'Do' replacement. And is also kind of weird: I
expect code to flow top to bottom when executing statements. To flip it around
is weird.

Let's say we just had some magic syntax for Unit@ _ <- ... ; ... syntax. We

On the topic of bind with (M@<A@>, (Unit@) { M@<B@> })
versus bind (M@<A@>, M@<B@>):

They are two different things. In the first case, the body of the function
isn't computed until explicitly asked for in the Do function. In the second
case, there is no body, the value is computed before the Do function is
called. For some kinds of M@ maybe that doesn't matter because it's free to
compute a function allocation, but other kinds makes a big difference. We
should not pretend we are doing one when we are doing the other. In other
words: don't make a special syntax for Unit@ _ := style binding. Be explicit
about that.

General thoughts on syntactic sugar:

The main goal is to take something that requires nesting and turn it into a
straight list. For example, instead of: { ... { ... { ... { ... }}}}, we
switch to: { ...; ... ; ...; ...; }. Instead of (..., (..., (..., (...)))), we
could switch to (..., ..., ..., ...), or something like that. That's what
gives us the syntactic wins.

Current use: function bind, case select.

You could imagine doing it for function application, in an expression or
statement context. For example:

  <expr> $ <expr>

Could be sugar for:

  <expr>(<expr>)

And <expr> $$ <stmt> could be sugar for <expr>({<stmt>}).

I don't know how useful those would be in practice. We know of a couple
possible uses:

  SkipTest(Str|'not good to run this test') $
  Test(Str|'mytest', (Unit@ _) { ...
  }),

  Do_(Put(Str|'hello')) $$

This could be useful in general. It could help with a special case Do, but not

Anyway, for such a rare case, the syntax isn't justified. Let's discuss the
more common case that I'm using now.

      AssertStream|False(/Std/Bool/Show%.Show(False));

The dollar sign was a nicer seperator between the test name and the value.
