Object Oriented
===============

I mean something very specific when I say this. Let me say what I mean and not
worry about the terminology for now.

The problem:

See thoughts/fble-0.6/stdlib.random.txt for context and motivation.

We have some core api, like generating random bits or writing a character to
an output stream. We want to build up things on top of those core APIs, like
generating random booleans, random integers, drawing from continuous
probability distributions. Or like writing lines and whole files to an output
stream.

The way I've been doing things for this so far has something like:

  # The core API:
  <<@>@>@ Rand@ = <<@>@ M@> { *(M@<Bit@> bit); }

  # A higher level function:
  <<@>@ M@>(Rand@<M@>, Monad@<M@>)<@ A@>(List@<A@>) { M@<List@<A@>>; }
  Shuffle = ...;

Whenever I want to call the higher level function, I have to provide M@,
Rand@<M@>, and Monad@<M@> all explicitly. Note that type inference cannot
infer the type M@ from Rand@<M@> or Monad@<M@>, because you can't infer a
general M@ type from a few concrete M@<X@> examples.

This leads to slightly annoying code like:

  (List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = {
    <@>@ M@ = Prng.M@;
    % m = Prng.m;
    <@>% r = Prng.r;
   
    (List@<Pos@> snake) {
      Int@ row := m.do(Range<M@>(r, m, MinRow, MaxRow));
      Int@ col := m.do(Range<M@>(r, m, MinCol, MaxCol));
      Pos@ pos = Pos@(row, col);
      InPosS(snake, pos).?(false: m.return(pos));
      GetNextFood(snake);
    };
  };

In particular, note the need to 'import' M@, m, r and the need to pass M@, r,
and m to the high level Range function.

I think we might be able to solve all of this if we take a slightly different
approach to defining our interfaces and add another variation of function
application syntax to fble.

The key ideas are:

1. If the Rand@ interface had M@ as a field and included the Monad@<M@> in its
interface, then type inference should be able to infer M@ and we only have to
pass one object 'r' instead of the three M@, r, m everywhere.

2. For whatever strange reason, it's much easier and more natural for me to
write things like:
   r.Range(Min, Max), r.Foo(a, b)

Than it is to write things like:
  Range(r, Min, Max), Foo(r, a, b).

This is what I mean by object oriented: the ability to do obj.Foo and have it
turn into a function call that takes obj as the first this/self argument.

The syntax proposal to start feels hacky / weird, but maybe also what we want:

  expr '.' name

Here name needs to be the name of a variable of function type. This is
syntactic sugar for function application:
  
  name(expr)

I have concerns with this syntax:
* It conflicts with struct and union access. But maybe we can say there is a
  priority: try struct or union access first. Only if those fail do we look
  for the function approach. Maybe that's even a feature: the object in
  question can provide a specialized implementation of a high level function
  if it wants to.
* Do we want/need the ability to apply extra arguments?
  - So we could do things like: obj.Foo() with zero additional arguments?
  - To allow for additional type inference based on those extra arguments?

Anyway, assuming we work that all out, my sample code from above simplifies to
something like:

  (List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = {
    Rand@ r = Prng.r;
   
    (List@<Pos@> snake) {
      Int@ row := r.do(r.Range(MinRow, MaxRow));
      Int@ col := r.do(r.Range(MinCol, MaxCol));
      Pos@ pos = Pos@(row, col);
      InPosS(snake, pos).?(false: r.return(pos));
      GetNextFood(snake);
    };
  };

Which feels surprisingly nicer.

Or, if you prefer, it's now much easier to write things like:

  <<@>@ M@>(Rand@<M@>, List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = <<@>@ M@>(Rand@<M@> r, List@<Pos@> snake) {
    Int@ row := r.do(r.Range(MinRow, MaxRow));
    Int@ col := r.do(r.Range(MinCol, MaxCol));
    Pos@ pos = Pos@(row, col);
    InPosS(snake, pos).?(false: r.return(pos));
    GetNextFood(snake);
  };

And then later on call it using:

  Prng.GetNextFood(...)

Even better.

That's the core idea anyway. I think there's lots of issues to work through
before I can feel good about it.

---

Here's what I suggest:
1. Try to come up with a different syntax than '.' for this:
 - To avoid ambiguity between member access and self func apply.
 - To more clearly convey the cost involved of calling a function.
2. Don't worry about function application type inference stuff.
 - Just: 'expr .$ name'  as the entire thing, it applies the function with a
   single argument.
 - We can internally merge it with subsequent applications if we want. Err...
   I guess we have to be clear from the spec point of view though?
3. Not sure about making it easier to 'subclass' struct types.
 - Is it worth it, or fine to manually inline, or fine to use the namespace?

Let's explore some syntax then.

  r.Range(MinRow, MaxRow)
  r.$Range(MinRow, MaxRow)
  r.:Range(MinRow, MaxRow)
  r:Range(MinRow, MaxRow)
  Range(r, MinRow, MaxRow)

  r | Range(MinRow, MaxRow)

  r..Range(MinRow, MaxRow);

  Int@ row := r.>do(r.>Range(MinRow, MaxRow));
  Int@ col := r.>do(r.>Range(MinCol, MaxCol));

  Int@ row := r.$do(r.$Range(MinRow, MaxRow));
  Int@ col := r.$do(r.$Range(MinCol, MaxCol));

  Int@ row := r/do(r/Range(MinRow, MaxRow));
  Int@ col := r/do(r/Range(MinCol, MaxCol));

  Int@ row := r.do(r:Range(MinRow, MaxRow));
  Int@ col := r.do(r:Range(MinCol, MaxCol));

  Int@ row := r .$ do(r .$ Range(MinRow, MaxRow));
  Int@ col := r .$ do(r .$ Range(MinCol, MaxCol));

  Int@ row := r.do(r.Range(MinRow, MaxRow));
  Int@ col := r.do(r.Range(MinCol, MaxCol));

  Int@ row := r,do(r,Range(MinRow, MaxRow));
  Int@ col := r,do(r,Range(MinCol, MaxCol));

  Int@ row := r.do(r..Range(MinRow, MaxRow));
  Int@ col := r.do(r..Range(MinCol, MaxCol));

  Int@ row := r.:do(r.:Range(MinRow, MaxRow));
  Int@ col := r.:do(r.:Range(MinCol, MaxCol));

  Int@ row := r.do(Range.r(MinRow, MaxRow));
  Int@ col := r.do(Range.r(MinCol, MaxCol));

  Int@ row := r.do(r::Range(MinRow, MaxRow));
  Int@ col := r.do(r::Range(MinCol, MaxCol));

  Int@ row := r.do(.r.Range(MinRow, MaxRow));
  Int@ col := r.do(.r.Range(MinCol, MaxCol));

  Int@ row := r.do(Range(r)(MinRow, MaxRow));
  Int@ col := r.do(Range(r)(MinCol, MaxCol));

  r^Range(MinRow, MaxRow)

  r:Range:(MinRow, MaxRow)

Sadly I can't come up with anything that looks right besides r.Range.
Considerations:
* It needs to clearly bind Range to r. Like: (r.Range)(x, y, ...), not 
   r.(Range(x, y, ...))

  x.$y.$z.$f
  x | y | z | f
  x . y(a, b)
  x.y(a, b)
  x.F(a, b)
  x..F(a, b)
  x.:F(a, b)
  x:F(a, b)
  x:y(a,b):z:f(w)
  x.y(a,b).z.f(w)
  x|y(a,b)|z|f(w)
  x.$y(a,b).$z.$f(w)

The trouble with '.$' is it doesn't feel like it binds tightly. It feels like:
x.$(f(a, b)) instead of (x.$f)(a, b)

  <<@>@ M@>(Rand@<M@>, List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = <<@>@ M@>(Rand@<M@> r, List@<Pos@> snake) {
    Int@ row := r.do(Range(r, MinRow, MaxRow));
    Int@ col := r.do(Range(r, MinCol, MaxCol));
    Pos@ pos = Pos@(row, col);
    InPosS(snake, pos).?(false: r.return(pos));
    GetNextFood(snake);
  };

  <<@>@ M@>(Rand@<M@>, List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = <<@>@ M@>(Rand@<M@> r, List@<Pos@> snake) {
    Int@ row := r.do(r.Range(MinRow, MaxRow));
    Int@ col := r.do(r.Range(MinCol, MaxCol));
    Pos@ pos = Pos@(row, col);
    InPosS(snake, pos).?(false: r.return(pos));
    GetNextFood(snake);
  };

I don't know. I'm not convinced r.Range buys us so much over Range(r). But
they are fundamentally different, right? In terms of pipelining things?

Consider this example:
  
  x.y(a,b).z.f(w)

That turns into:

  f(z(y(x, a, b)), w)

The ordering is totally different. I like data flowing left to right with the
'.' operator going through functions.

But it's different from pipe

  x|y(a,b)|z|f(w)

Which feels much more like

  x | y(a,b) | z | f(w)

Rather than
  (((x.y)(a,b).z).f)(w)


  x~y(a,b)~z~f(w)

  <<@>@ M@>(Rand@<M@>, List@<Pos@>) { Prng.M@<Pos@>; }
  GetNextFood = <<@>@ M@>(Rand@<M@> r, List@<Pos@> snake) {
    Int@ row := r.do(r~Range(MinRow, MaxRow));
    Int@ col := r.do(r~Range(MinCol, MaxCol));
    Pos@ pos = Pos@(row, col);
    InPosS(snake, pos).?(false: r.return(pos));
    GetNextFood(snake);
  };

Actually, '~' isn't bad. Looks kind of like -> from pointers:
  r->Range(...)
  r~Range(...)

Maybe let's propose that. I'm not using ~ for anything else right now, right?

Or is that bad for other reasons, like keyboard ergonomics, other keyboard
layouts, etc?

r~Do(r~Range(MinRow, MaxRow));

My guess is I would get used to ~ if I started using it for a while.

f(x, y)~Range(MinRow, MaxRow)

Okay. That feels a little weird.

f(x, y).Range(MinRow, MaxRow)

Dunno. Let it stew some more.

Next challenge: subclasses.

I want Rand@<M@> to have a Monad@<M@> instance built in. Options:
1. Composition. r.Monad gives us the Monad@<M@>.
 - Easy to construct.
 - Easy to pass subinterface around.
 - Annoying to have to refer to everywhere
     r.Monad.do(...)
     r.Monad.return(...)
   Compared to:
     r.do(...)
     r.return(...)
 - Composes well with arbitrary nesting of different interfaces.

2. Inlining. r has fields do and return.
 - Slightly awkward to specify and construct.
 - Slightly awkward to pass around subinterfaces.
 - Nice to refer to everywhere.
 - Doesn't compose well with arbitrary nesting of different interfaces.

Composition would be really nice if we could get around the name issue. But
this is like having object oriented name resolution. Like, what if we say r.do
first looks in 'r', then starts to look in subfields of r. Like how we do with
literals now. Just hope there isn't any conflict. If there is conflict, you
can explicitly specify the one you want, and we can have a default order
otherwise.

In short, feels like I should use composition, but maybe provide some compiler
magic to allow us to shorten references to subfields.

---

Another challenge that comes up with including Monad@<M@> in the subinterface
for Rand@<M@> is what happens when we want to compose, say, Rand@<M@> and
State@<M@>, or whatever other high level interface. If everyone carries around
Monad@<M@>, how do you know they are the same? Isn't it weird to have copies
of it everywhere?

It's almost like I want two different layers: the raw interfaces and the
composition of interfaces. Mix and match that way.

For example:
  Rand@<M@> has choose/feed.
  State@<S@, M@> just has get/put.
  Monad@<M@> has do/return.

Instead of passing around Rand@<M@>, Monad@<M@> always as pairs, we pass
around something else. Called??? Maybe we call the above RandI@, StateI@,
MonadI@? Hmm... Anyway, whatever we call this top level thing, it has M@,
Rand@<M@>, and Monad@<M@>.

Consider for example the Range function:

  # Generates a random integer uniformly in the range [lo, hi]
  <<@>@ M@>(Rand@<M@>, Monad@<M@>, Int@, Int@) { M@<Int@>; }
  Range = <<@>@ M@>(Rand@<M@> r, Monad@<M@> m, Int@ lo, Int@ hi) {
    Int@ x := m.do $ Of(r, m, Incr(Sub(hi, lo)));
    m.return $ Add(lo, x);
  };

Instead of this interface... maybe:

  <<@>@ M@>(RandM@<M@>, Int@, Int@) { M@<Int@>; }
  Range = <<@>@ M@>(RandM@<M@> this, Int@ lo, Int@ hi) {
    Int@ x := this.m.do $ Of(this, Incr(Sub(hi, lo)));
    this.m.return $ Add(lo, x);
  };

That's not too bad. If we had magic syntax, it would be like:

  <<@>@ M@>(RandM@<M@>, Int@, Int@) { M@<Int@>; }
  Range = <<@>@ M@>(RandM@<M@> this, Int@ lo, Int@ hi) {
    Int@ x := this.m.do $ this~Of(Incr(Sub(hi, lo)));
    this.m.return $ Add(lo, x);
  };

I don't think the magic syntax helps here, to be honest. I suppose the nice
thing here is having a generic otherwise meaningless name 'this'.

Maybe let's see how far we can go without adding magic syntax. Just use what
we have already today. In other words, for my 3 main challenge:

1. Don't have special syntax. Use normal function application. No ambiguity to
worry about, no new syntax needed.

2. No need to worry about applying extra arguments. Not applicable.

3. Subclass by having raw interfaces be separate from top level interfaces.
The top level interface is just a small composition with short names.
