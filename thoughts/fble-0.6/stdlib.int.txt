I'm providing unbounded integers rather than fixed width, so the bitwise

Go:

Int is the first somewhat complex library to provide. It's worth taking some
time to plan it out in more detail. And compare more with other, existing
libraries.

High level: I want an unbounded, signed integer type. Don't include bitwise
operations.

Minimum needed to work with ints in practice:
* Int@ type, abstract.
* Some way to construct any Int@ value:
 - 0, Neg, Add1, Mul2
* Some way to deconstruct any Int@ value:
 - IsZero, IsNegative, IsPositive, Div2

What are the questions / decisions to make here?
* Int@ type is clear.
* Literals: how many to define in the language?
  - Just 0? -1, 0, 1? 0 through 10?
* Unary operators: Incr, Decr, Neg, Abs, Sgn?, Sqrt, Div2?, Exp2
  - Sgn is an interesting one. Options:
    a. Have IsZero, IsNegative, IsPositive, only. No Sgn.
    b. Have Sgn go to '-1', '0', or '1' only. No Is...
    c. Have Eq, Lt, Gt, etc. only, no IsZero, IsNegative, IsPositive, Sgn.
  - Should we have Div2? Corresponds to shift?
    Or Div 2^n? For more general shift right?
    Or Mul 2^n? For more general shift left?
* Binary operators: Add, Sub, Mul, Max, Min, Div, 
  - Do we want general a^b exponentiation? Why not?
* Comparison operators: Lt, Gt, Le, Ge, Eq, Ne
  - Do we need Ne if we have Eq?
* Show: Read, Show. But that's an add on I would say. I can consider that
  later.

Additional things Java has that may be worth considering:
* Nothing relevant. The above catches it all.

Let's assume I've at least mentioned here all the relevant functions to
include for integer to start. How to decide on some of these things?

Example: Incr.
  Incr is the same as Add(x, 1). I think it's worth having still, because it's
  a common operation, and Incr as a function is simpler, and potentially more
  efficient.

Literals:
  I know eventually I'm going to support literals via Digit@, Char@.
  I expect most people to use those forms.
  
I want general literals for: Decimal and Hex for sure. Binary and Octal too?
But I can iron out those details later. The question is, what do we want to
provide outside the context of those literals?

I like: 0, 1, 2^n, -x. So, yes for 0 and 1. But no for anything else.

More on which kinds of literals:
* Decimal and Hex for sure. No question. That matches C99.
* Octal is in C99, but I personally have never used it. Is it obsolete?
* Binary seems useful, but pretty rare? It's not in C99.

Go supports octal and binary. It can't hurt. Let's just have Oct and Bin as
well.

We can decide later the precise syntax for the literals to support. +-? _ for
groupings? I vote we support '-' to negate, and some sort of separator,
because I don't see how adding those can hurt anything.

Next, let's see if we can clean up naming and population of operators.

* Incr - Worth having, but what to call it?
  Incr, Add1, '++'
  I like Incr. I use it a bunch, definitely worth having.

* Neg - worth having, anything else to call it?
  Minus? I guess Neg is short for negate?

Math calls negation: additive inverse, because when added to a number, it
results in the additive identity. Opposite, negative, arithmetic negation.

Let's go with 'Neg', it's natural to me.

To what extent do we want to divide up the type into different modules?

Existing division:
  /Core/Int%
  /Core/Int/Div%
  /Core/Int/Sqrt%
  /Core/Int/Eq%
  /Core/Int/Lit%

I like dividing things up for implementation purposes for sure, to have
smaller context to work with. But we can divide things up for implementation
and then combine them all together for public export.

Previously I would have said we should separate Eq% because it depends on
Bool% while the others don't. Do I still buy that argument?

  /Core/Int%: Unit, 
  /Core/Int/Div%: Int, Undef
  /Core/Int/Sqrt%: Int, Lit
  /Core/Int/Eq%: Int, Bool
  /Core/Int/Lit%: Digits

Digits, Undef, Bool, and Unit are all really small. But that's because we are
at the base of things. Digits I'll probably inline.

Presumably Div and Sqrt and Eq are a bit hefty, and if we want to use Int
without those, we could save space by not having to compute them during
program startup and not having to store them in memory.

When we standardize an Eq class, we might want to naturally have a separate
/Core/Int/Eq% module like we will for /Core/Bool/Eq%, and /Core/Bool/Show%,
and /Core/Int/Show%, because Show depends on Char depends on Int.

It can be hard to find where to look for things if we have all these
submodules. Like, I want to know if a particular method exists, say String
split. I don't know which file to look for that necessarily of there are a
bunch of different files per conceptual type.

We could also chose to specify a breakdown of modules, and a higher level
module that contains everything in the breakdown.

Separating into submodules is kind of like saying we have an optional module.
It also would allow for optional modules. You don't get this by default, but
you can add it extra if you like.

Practically speaking, the competing considerations are:
* How hard to remember which module defines what you want?
* How much extra unused stuff do you end up pulling into your program?
