Stdlib Env
==========

Goal: Be able to access environment variables.

Conceptually:
* At program startup we are provided with a map from strings to strings which
  is the environment.

C99:
* getenv

Posix:
* getenv, setenv, unsetenv

Setting environment variables is relevant for forking child processes with new
environments. Which we don't support yet, but perhaps could one day.

Many languages, such as python, treat environment operations as separate from
the underlying posix environment operations. I'm not sure why. Based on online
discussions, some possibilities:
* getenv/setenv calls are too slow. They cache it for performance.
* getenv/setenv doesn't give a way to iterate over the environment variables.
* They don't want calls to C or other foreign functions that change
  environments to be visible in python code.
* Concerns about multithreaded access to the data.

See https://discuss.python.org/t/method-to-refresh-os-environ/54774 for some
interesting discussion.

Also: https://www.evanjones.ca/setenv-is-not-thread-safe.html

Hmm... in summary:
* getenv/setenv are not thread safe. You can't call setenv in a multithread
  program safely. See 'MT-Unsafe' in the man page.

It sounds like getenv is implemented internally as a linear search in glibc.

What to take away from this?
* If anyone calls setenv in a multithreaded program, we are screwed. We can't
  read a single environment variable safely. We can't read the list of
  environment variables safely.
* It's probably not too much more expensive to read the whole environment at
  once into fble than to call getenv. At least in terms of order of
  complexity.

There is some circular motivation going on here.
* My own motivation is only for getenv. I have no desire for setenv. That only
  makes sense to me if you are spawning subprocesses to have a way to pass a
  modified environment, but we could provide other ways of doing that.
* But I fear FFI calls could be done that call setenv, which then raise the
  question of whether those are visible to getenv.
* Assuming getenv is faster than iterating over the entire environment, then I
  would want to implement getenv via the posix library function, but that sees
  effects of whoever calls setenv.
* If it sees effects of whoever called getenv, then getenv has to be
  sequenced.
* If getenv is sequenced, that only makes sense conceptually if there is an
  ability to make changes to the environment, which it is also sequenced with.

Here is what I'm going to propose conceptually:
* Add a pure function as another argument, in addition to List@<String@> args,
  to Cli Main@. We take a snapshot of the environment at startup. There is no
  way to observe changes made via setenv.

Important points here:
1. We do not support any form of setenv from within fble.
2. We don't give the user any control over when the environment is read. That
   way we can do it in a hopefully safe manner at the start before anything
   messes things up by calling setenv. Though no guarantee of that.

Let's remember that C code can mess anything up any time it wants. Let's
assume people are good citizens and do not call setenv from multithreaded
code. Let's avoid support for setenv from within fble to avoid that being its
own source of multithreaded setenv calls.

Since the intention is for the environment to be constant, it doesn't make
sense to provide a way to 'refresh' the environment or ask what it is now. It
conceptually only makes sense to fetch it at startup.

Hopefully the cost isn't too high? It's a little sad if we have to parse a
bunch of environment variables for applications that never query the
environment.

An alternative would be to provide native function to read the current
environment into an fble function. Arrange for that to mostly only be called
by apps that want to read some environment function.

I guess this means I want apps to be able to iterate over the list of all
environment variables, not just query ones they know the name of? Depends. We
could do the API either way.

Let's not worry about how we will implement things. Let's think about what
interfaces could make sense.

1. (String@) { Maybe@<String@>; }
Constant, read only, no way to iterate through environment to discover what's
there, but if you know what you are looking for you can check for it.

2. Map@<String@, String@>
Constant, read only, but with an option to discover what's there.

3. (String@) { M@<Maybe@<String@>>; } get,
   (String@, String@) { M@<Unit@>; } set,
   (String@) { M@<Unit@>; } del
Dynamic. You can set or remove values which are seen on subsequent get.
Everything is sequenced via M@. No way to iterate through environment to
discover what's there.

4. M@<Map@<String@, String@>> get, (Map@<String@, String@>) { M@<Unit@>; } set
Dynamic, with the ability to iterate through the environment to discover
what's there.

So there's only two choices at this level:
1. Can you iterate or not.
2. Is it constant or not (?). Or, can you feed back to caller or not?

Here is my concept of the usefulness of an environment:
* It provides a way to pass things from a to c, via an intermediate node b,
  where b doesn't have to know what's being passed. That's pretty powerful.
* It provides a way to store a reusable configuration. For example, set up
  some arguments for FBLE_PACKAGE_PATH once, then everyone who cares about it
  can reuse that. Even if it's direct from a to b1, b2, b3, etc.

Honestly, I don't see a reason to allow apps to iterate over all values of the
environment. As long as they have some way to create a new environment that
references the unknown values of the previous environment implicitly.
Conceptually, when a is passing something to c via b, that shouldn't
necessarily be exposed to b? On the other hand, maybe b wants to have the
option to pick and chose which of the environment values are okay to pass to
c. It can't filter things out unless it has some way to discover what's there.

The more general approach would be to all an app to iterate over all values of
the environment.

I guess the other question is: can you modify the environment of your caller
or not. It makes sense to control what your children see in the environment.
But if we want to delegate that to some helper function, it also makes sense
to be able to return a modified environment to your caller. That way the
helper function returns the modified environment to the parent, which can then
pass that to the child.

4. M@<Map@<String@, String@>> get, (Map@<String@, String@>) { M@<Unit@>; } set
Is a State@<Env@> kind of interface. That sounds like a common kind of
interface for things to have in fble. Env@ is special because it provides a
mapping with user defined and pre-agreed-upon keys, and it represents all data
as Strings.

You could think of a file system in a similar way, but using Stream@ instead
of Stream@ for holding the data.

What if we abstracted that notion of a registry out? You have a registry of
keyed values. You can get a value. You can set a value. Maybe you can iterate
over the list of values.

What would (4) look like if that's what we wanted to support via native code?
* It's tedious from a performance point of view that you have to read and
  write entire environments at a time. It's not a problem as pure fble,
  because the caller has already written it and you kind of do an atomic swap
  which is free. As soon as you have to marshal to and from C, it gets more
  expensive. We could solve that with a more abstract set/get/unset interface.

If I remember correctly, posix lets you modify environ, and subsequent calls
to getenv/setenv are supposed to adapt to that. Maybe we could implement
getenv and setenv as get/set this way? Get: iterates over environ, trusting
that nobody is going to modify it in the mean time. Set: create a new
environment, then manually swap things over.

Two uses of getenv in my programs today, both the same use case: satgame,
invaders.
* Invaders: $HOME/.fble-invaders is location of high score file.
* SatGame: $HOME/.fble-satgame is location of random seed file.

How do I want to convey these to my abstract application? There are a bunch of
different options:
* Pass in ENV. Let them decide $HOME is what they want.
* Pass equivalent of $HOME as an additional command line argument.
* Pass equivalent of $HOME as an extra argument to the function.

I think, conceptually, what these apps want is the name of a directory
associated with a particular user that will persist across runs. Again, this
is a special case for games. I expect other tools would use the environment
differently.

It kind of gets at the question of typed versus untyped interface for an app.
Command line args as strings represents untyped. Maybe we should pull out
typed explicitly separately, where command line args are already fully parsed.
That way it makes sense to pass an explicitly argument for the prefix of files
associated with the user that are owned by this game.

Then, in the untyped interface, maybe access the environment variable $HOME
directly right there.

But it gets confusing. Why should the typed interface take a name of a file
that it then has to read? Why wouldn't it take the already read contents of
the state and return the new contents of the state? Abstract away all platform
specific stuff in the untyped interface?

But then, what if I want to test that untyped interface? Then we still want to
keep it abstract and have yet another layer mapping that to the native
interface?

It feels like I want the native interface to be as small as possible, because
that's the bit that we can't test within fble. If we go that route, then
shouldn't I pass Env@ directly to the untyped main program, have that do all
the mapping from args, env, stdio to content, and then it calls on?

I guess I'm proposing three layers:
1. Native layer. As small as possible. Essentially lists the native interfaces
needed and passes them directly to the untyped layer.
2. Untyped layer. Adapts native layer to typed layer. Can be tested as pure
fble, because it doesn't depend on native. But has knowledge of how to
interact with the native world: e.g. via environment variables, command line
args, and files.
3. Typed layer. Completely separate from command line. Takes the typed
information it really wants (e.g. previous high score), returns what it wants
saved (e.g. new high score). Can be tested as pure fble. Presumably this is
the most platform independent code.

Sound reasonable? Let's give it a try. Maybe structure like this first, then
do my get env swapover.
