For something like Eq@, Ord@, etc. I think we can categorize methods into:
minimal, core, and other. How exactly to organize is unclear, but for example:

minimal: Ord
core: Ord, Lt, Le, Eq, Gt, Ne, etc.
extended: Sort, maybe Max, Min?

We'll always want extended, and those should always be implemented by taking
an Ord@ instance as an argument.

We'll always need minimal. It could be different for different things though.
Something might have Le as minimal, something else Ord.

What matters for real?
* What we pass to extended functions: minimal or core?
 - Performance implications: if there are some core functions that we want to
   specialize for performance reasons, extended functions can only get access
   to those if we pass core rather than minimal.
* Whether we call a generic function like Min passing Ord, or whether we have
  a specialized version for a particular type, e.g. Int Min, or both.
 - This doesn't care about minimal vs. core vs. extended. You can easily have
   a generic implementation and provide a type specific specialized
   implementation.
* Whether we define a group of functions all at once with a single minimal (or
  core?) arg, or we pass the Ord@ arg to each individual function.
 - For example, we could have a single function to produce the set of core ord
   functions from the minimal ord function, and have that be much easier to
   reuse than having to pass the same minimal ord thing to all the individual
   functions.

Things will get more interesting with the Map@ interface, which parameterizes
both the element type and the container type. In that case there is no clear
'minimal' implementation, it really depends heavily on the element and
container types.

Here is what I'm leaning towards:
* Have an interface for the core methods of something.
* Have helper functions to implement the interface from the minimal methods of
  something.
* Group together other methods as feels reasonable, taking the interface value
  for a group of methods at a time rather than one by one.

Anyway, let's get down to details. The questions are:

A. Ord@ as just a compare, or as a core set of cmp, eq, lt, min, max, etc?
B. Generic implementation of eq, lt, min all bundled together at once, or
   separate?

Benefits of just compare:
* No dependency on Bool@.
* Easy to implement.

Benefits of core set:
* Can provided 'optimized' implementations for eq, lt, min, max, etc. rather
  than force use of the generic implementations.

Benefits of bundling generic all together at once:
* Easy to implement the whole set with just comparison.

Benefits of separating them:
* No need to allocate a whole bunch of functions one-off if you just want one
  of them.

I kind of feel like:
* If using just compare, we want separate implementations of generics.
* If using core set, we want bundled implementation of generics.

The real concern from a performance point of view is if we are given Ord@ as
an argument to a generic function (e.g. list sort), and we want to use a
different core set function in the implementation fairly heavily (e.g. le).
Specifically:
* If we can't use the optimized version of le, seems a shame.
  But really how much more optimal can you make le compared to cmp?
* If we have to recreate the generic bundled version every time the function
  sort is called, that sounds potentially expensive?

Performance suggests we want as much specializable pre-allocated functions as
we can get. That suggests passing around the core set when implementing
generic functions.

I would say, certainly Eq can be implemented more efficiently than compare on
average. We know right away that an odd number doesn't equal an even number,
but we might have to do a bunch more work to tell which is greater.

Proposal:
* /Std/Eq% is as we have it today.
* /Std/Foo/Eq% is as we have it today
 - Eq implements Eq@, it's the equality function.
 - Optional Ne implements not equals, only for types we think we'll want to
   call it fairly often from the top level like Int@.
* /Std/Cmp% defines Cmp@ as it does today: the comparison function only.
 - But also defines generic Lt, Le, Gt, Ge, Max, and Min functions.
 - And also a helper function (named 'Cmp'?) from Cmp@ to the set of
   functions: Cmp, Eq, Ne, Lt, Le, Gt, Ge, Max, Min.
* /Std/Foo/Cmp% implements Cmp@
 - And exports as much of the top level Lt, Le, Gt, Ge, Min, Max that we might
   want for the type, such as Int@.
* If a function wants fast equality, it should take Eq@ as an argument.
* If a function wants Cmp/Lt/Le/Gt/Ge/Max/Min, it should take Cmp@ as an
  argument.
* If a function wants Cmp and fast equality, it should take both Eq@ and Cmp@
  as arguments.
  - Because Eq@ implementations can be fundamentally faster if implemented
    directly than via Cmp, because we only need to find one difference rather
    than traverse the entire value.

Miscellaneous thoughts:
* Don't worry about /Std/Int/Sign% type. It can be separate. That's fine.
* Fundamentally, Cmp is a total order, whereas Lt/Le/... could be partial
  orders. Don't bother with the distinction. We can make everything totally
  ordered. Probably better aligns with expectations that way anyway.
* Xor is Boolean Ne. XNor is boolean Eq.
  But it's probably still worth defining Xor, because that's a common name to
  want to use.
* Import expressions should make it more convenient to define and re-export
  wrapper functions like Lt/Le/Gt/Ge/...
