Fble 0.6 TODO

 - Change getenv function to a pure env function.

* Add test for StrE with \q escape.
* Add spec tests for foo $ bar syntaxes.
  - right associativity
  - works in expr and stmt context

* Update tutorials for new bind and function application syntax.

* Print function names along with error location on stack trace?

* Optimize /Std/Int/Show% implementation by pulling out int values to top
  level rather than calling Int|* over and over?

* Add function to show an integer base 16? Base 2? Base anything?

* Change /Core/Stdio/StringO% to take Output@ instead of String@.
* Change /Core/Debug%.TraceLn function to take Output@ instead of String@?
* Double check if I should be able to write:
   @ ShowM@ = /Core/Show%.ShowM@, when ShowM@ has kind <@>@.
  Seems like I can't, but I thought I made it so I could.

* Figure out why we see the same errors reported many times when we build.
 - And fix it.

* Get rid of /Std/String/Format% ?
 - Prefer the Show@ version of Format?

* Cover all escape chars in StrE tests.
* Better implementation of ReadHex in /Std/String/Escape.
* Replace uses of 'Concat' with 'Format' where appropriate.
* Add vim syntax highlighting for \ and % escape sequences?
* Double check @module[...] values are correct in /Std/...*.fble files.

* Double check performance of ReverseComplement, specifically the
  ComplementCode function. Maybe use a map instead?
  Similar for /Invaders/Graphics%.String.

* Change FbleNewCharValue to take wchar_t.
* Change FbleCharValueAccess to return wchar_t.

* Does private type access restrict access to literals?

  adding a spec test for.

* Optimize encoding of FbleNewLiteralValue prgm?

* Add a 'function' standard library?
 - compose, flip

* Update tutorials for string literals: Foo(Str|...) now instead of Foo|...;
* Update tutorials for character library.
* Update tutorials for use of Concat instead of Strs.

* Implement Cmp@ for Unit, Bool, Int, Char, List.

* Figure out how to properly document List@ type

* How to document a poly? @func? @value?
* How to document methods of interface structs?
* How to document an interface function type, like Eq@?
* /Std/List/Eq% docs could be more precise and include examples.
* How to document internal functions, e.g. /Std/List/Sort%.Insert?
  Make sure this doesn't get shown in reference documentation.
  e.g. /Std/Int/Lit% internal helper functions.

* Should I add @example as a doc comment field? Seems useful?
* Reorder arguments in body function passed to ForEach?

* Restrict Int@ type internals to /Std/Int%.
* Restrict /Std/Int/IntP* modules to /Std/Int%.
* Update references to /Core/Int in tutorials.
* Update references to /Core/List in tutorials.

  or not.
  And add spec tests for that case.

* Rename /Core/Unit% to /Std/Unit% in tutorials, if there are any references.

* Revamp the standard library.
 - Redesign from the ground up, looking at existing standard libraries for
   guidance.
 - Test and document (and tutorialize?) all functions.
 - Implement rosettacode.org tasks under pkgs/rosetta to test it out.

* Optimize C and interpreter backends like we did aarch64?
  By inlining logic to access fble values instead of calling library
  functions.

* Design a package management / versioning scheme?
 - see thoughts/fble-0.5/modular_compilation.txt for context.

* Clean up program loading
 - see thoughts/fble-0.5/modular_compilation.txt for context.
 - Clarify the spec with respect to builtins referring to other builtins and
   the recursive module requirements in case of difference between builtins,
   .fble files, and header files.
 - Add tests for:
   a. Try to compile a builtin module.
   b. builtin A -> builtin B, .fble C -> builtin B.
   c. builtin A, A.fble.@ -> B.fble, B.fble -> /C%,
      C.fble has bugs, C.fble.@ doesn't. This should not fail to execute.
 - Update the implementation to pass the tests.
 - Consider separating the FbleModule* for execution and for type for a given
   module path.
 - Add option for loading modules from .so files?
 - Have fble-stdio check that the function has the right type?
 - Load minimum dependencies needed for type checking when possible?
  
* Make programs run twice as fast.
 - Use native stack for stack allocations.

 - Specialize call logic based on what we can figure out statically in terms
