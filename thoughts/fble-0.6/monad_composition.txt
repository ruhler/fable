Monad Composition
=================

This keeps coming up. I've had some thoughts and made some progress, but I'm
not convinced my current approach scales the way it should. So let's review.

I want to be able to write functions that work for any monad that supports a
specific interface. For example, any monad that supports random number
generation. Then I want to be able to reuse those functions with a variety of
different monad instances easily: a pure state monad, an Io monad, a
transformed Io monad, a monad that can also do input and output, etc.

See thoughts/fble-0.6/stdlib.random.txt for discussion of various approaches
and explanation of how I got to where I am today. The progression is roughly:

1. Write the function to use a concrete monad.
 - But then it doesn't compose well with variations on that monad.

2. Write a function that works forall M@ that have a Monad@ instance and an
Rng@ instance. Rng@ is separate from Monad@ in this case.
 - But then type inference doesn't work to infer the type M@ and we have to
   pass around 3 arguments everywhere: M@, m, r.

3. Write a function that works forall M@ and takes an RngM@ instance, where
RngM@ has an explicit M@ field to enable type inference of M@ and includes a
Monad@ subfield so we can pass around one argument for everything we need.
 - But now any time we want to say m.do or m.return, we now say something like
   rng.m.do, rng.m.return. Is that better than (2)?

I'm worried this doesn't scale up too well though. Like, what if I want to
compose an Rng@ with Io@. Am I given RngM@ and IoM@? Or I have to define a new
RngIoM@? How can I extract easily an RngM@ from this RngIoM@ to pass to a
function that only wants the Rng@ part of the interface?

Can we include the M@ field in all sub-structures so there is no need to
distinguish between Rng@ and RngM@? How about the Monad@ field? Any concern
about having two different named Monad@ subfields?

Here's what I think the challenge is:

We want to define some set of axioms and parameterize a family of separately
defined functions by those axioms. Sample axioms: do, return, random, run.

To parameterize functions by the axioms, we pass the axioms as arguments to
the functions. Either individually, or grouped together in structures, or some
mix of approaches.

It's tedious to pass these things around everywhere:
* Annoying to have to access the axiom from whatever structure you pass it
  with, e.g. io.m.return, io.io.run
* Annoying to have to pass the axioms everywhere we want to call things, 
  e.g. Range(rng, m, Int|10, Int|20), Range(MkRng(iorng), m, Int|10, Int|20)

You reduce the cost of passing things everywhere by grouping them together
into one object, but that increases the cost of accessing the axioms.

How can we solve this?

First off, understand that the language as full support for making the names
of things nicer, if we are willing to use it. It's the same thing we do for
top level imports. Imagine writing at the top of each function body:

  % return = io.m.return;
  % do = io.m.do;
  % run = io.io.run;
  % Range = /Core/Int/Rand%.Range(rng);

Now the body of the function lets us write exactly what we want to write. The
only thing missing would be instead of saying things like Foo(a, ...), Foo(b,
..), we could write a.Foo(...), b.Foo(...), which for whatever reason I don't
fully understand, feels much nicer.

But top level imports are super tedious. I know it. Look at the above example.
I didn't bother figuring out the right kinds for things, because it's too much
effort when the compiler can tell you the right thing anyway.

What makes importing tedious:
* Having to figure out the kinds of things, when it feels like they don't
  matter at all in practice.
* Taking up so many lines: One line for each definition in practice. For a
  short function, this could easily take up as much space as the content of
  the function. It often takes up half of a module's definition just importing
  things.
* Module paths are long and slightly annoying to write and deal with.
* Notice how much duplication is in the above example: io is reused 3 times,
  return, do, run, Range all show up twice.

So here's the first potential solution idea: come up with a much nicer syntax
for importing things. That could help with monad composition here, but also
with top level module imports.

One challenge is that I really like the fact that you can search for any
variable name and jump to the definition. We don't ever define variables
implicitly. You can always track down where and how something is defined.

Example wild ideas for how to simplify top level definitions. Using the
examples from above, which capture a decent range of the different things I
might want:

  ^ io(.m(.return, .do), .io(.run));
  ^ /Core/Int/Rand%.Range(rng);

Is it much better? I'm not sure. Should I try using the above approach for
some of my functions, see how it feels? Key ideas:
* Skip writing the kind. The compiler knows that.
* Reuse names from the defining expression for the variable.
* Allow multiple imports from a single item at once.

  ^ @(m: @(return, do), io: @(run)) = io;

An alternative approach is to try to automatically figure out what someone
means when they refer to some name. Call it compiler magic. What kinds of
searches should we do?

* See if a name refers to a subfield of a variable in scope.
  For example, you say 'return'. That's not defined, but there is a local
  variable 'm', or 'io', where if we search, we find 'm.return' and
  'io.m.return' options for return.
* See if you can infer an argument for something.
  For example, you say 'Range', it takes an RngM@ as an argument, we happen to
  have an RngM@ in scope, so automatically pass that.

Might be nice, but also very scary to do that kind of magic. How do you figure
out what things are actually referring to?

Another style of approach could be have an: 'import everything from this
struct' syntax. Then the user could write helper functions. Actually, that's
pretty similar to the thing I was trying above.

  ^ io.m;   # Imports return and do fields of io.m.
  ^ io.io;  # Imports run field.
  ^ @(Range: /Core/Int/Rand%.Range(rng)); # Imports Range field

But we better provide a nice way to import a subset. And things start to get a
little messy.

Think about how some other languages do things.

C++:
* Classes give some fancy magic:
 - Member fields let you bundle together axioms associated with the class
   instance.
 - this.Foo() can be written just Foo: let's you pass all those axioms around
   implicitly.
* Essentially we are hiding the axioms in the class state, which is passed
  around explicitly and used for name resolution. And people can add functions
  to that class. But it's a closed list. If I was happy with a closed list, I
  could do everything that way too, pass around a single this option and use,
  e.g. this.do, this.return, this.random, this.run, this.Range everywhere.

Now I have a pretty good sense when writing a function what I'm taking as
import that is kind of an interface. I could easily separate out the interface
arguments from regular arguments. Let's say we have a notion of an interface
that you pass to a function. This is an environment. A set of name value
pairs.

When I call a function that expects some environment, let's say it inherits
from the current environment. And/or I can explicitly modify the environment
when calling that function if I want to override things.

Is it as simple as that? One key being: I can define more in my environment
than a callee requires, which is nice for composition. We try to put
everything at the top level of the environment, so I would have:
  do, return, random, run
instead of: m.do, m.return, rng.random, io.run.

When specifying your environment, you have to deal with name conflicts there.
So environments can maybe hold sub-environments.

In a sense, an environment is like a read only struct value, and we can pass
any struct value whose type is a superset of what is required by the user.

We could define an environment using a struct type. We could provide helpers
to inline things. For example:

  Monad@ = *(Do@ do, Return@ return);
  Io@ = *(Run@ run);
  Rng@ = *(Random@ random);
  IoM@ = *(Monad@ *, Io@ *);

Now I can have a function that expects Monad@, and I have an instance of IoM@,
and I can pass that IoM@ directly do the function where a Monad@ is expected,
because it has everything we need.

Now imagine we have a nice way to import everything in an environment. And
imagine we have a nice way to capture everything in the environment, or, we
pass the environment implicitly. Everything would be solved, right?

Experiment 1:

Experience report:
* Consider State vs. StateM too, in addition to Rng and Io.
* It's not particularly harder to construct an Rng than the previous RngM.
 - Just have to remember what order do and return are declared in.
* As a user of Rng, feels pretty nice.
* Biggest issue so far is annoyance of having to copy paste do and return
  declarations from Monad@.
* It's nice use Rng@ directly instead of RngM@.

I would say, for Rng@, this feels like an improvement. It's much nicer to use.
It's not any harder to construct. It's only very slightly annoying to define
because of duplicating do and return declarations.

* Reference methods via m.foo everywhere.

Experience report:
* Defining pure state monad instance is actually easier without the extra
  layers of indirection.
* 'P' is a weird variable name to use for a pure state monad.
* Maybe IStream@<M@> should include do, return too?
 - Which starts to get trickier, because I expect to have access to the monad
   type separately for that...
* I'm doing @(P.return, P.do) a bunch to extract the Monad@ implementation
  from a State@. Is that something I should be concerned about?
 - Actually, it needs to be Monad@(P.return, P.do).
 - Actually, it needs to be Monad@<P.M@>(P.return, P.do)
* Now I see cases where having M@ as part of Monad@ would be nice.
* When I want to create two separate state views for the same monad, in
  /Games/BenchmarkGame/Game%, things are getting messy.

Unlike the Rng case, I'm not at all convinced this is an improvement. What's
bad is, the Rng case would run into the same issues if it was used via as much
composition as StateM is being used in. For Rng@, it's like there's one
interface that everyone uses, and so it's fine. For State@, it's like we are
passing around lots of different variations of views of state monads to
different things, so we have lots of different instances that we want to
construct and pass around. That's making things unpleasant.

I'm going to back out this change. Go back to the drawing board. Use StateM@
as my driving example, because I clearly do more interesting things with that
than with Rng@.
