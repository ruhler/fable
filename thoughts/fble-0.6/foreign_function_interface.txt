Foreign Function Interface
==========================

Revisiting the question of having FFI baked into the language.

See thoughts/fble-0.4/foreign_function_interface.txt for background.

I'm thinking about it differently today though. The main goals for me with FFI
now are:
* Eliminate the need to write tedious, error prone code for defining profile
  blocks and FbleExecutables in my native modules.
* Allow easier mixing of foreign and pure code in the same module.

Concrete examples:
* I'm pretty sure I have my profile block ids wrong in my implementation of
  pkgs/core/stdio.fble.c, or I did, or I broke things and didn't notice.
* It's a pain to add new profile blocks and adjust everything based on
  position.
* It's a pain to define the module paths needed for the .path field of an
  FblePreloadedModule.
* It's a pain to have FooImpl and Foo wrapper function for every native
  function.
* For native getenv, I wanted to define Env@ type in the same module as
  getenv, and I wanted to provide a Lookup function that translated from the
  list of pairs to a lookup function in the same module.

Looking at things now, I'm not sure the details quite work out, but the idea
this time is what if we focus on only having to write the run functions for
things, and let the compiler do all the rest.

The idea is the compiler needs to know the function type, including number of
actual arguments, and the name of the symbol, e.g. FbleIStreamImpl. It writes
the wrapper function and profiling things as needed. Probably best to have the
syntax be a function type/call like thing:

<<@>@ M@>(Native@<M@>, Monad@<M@>) { M@<IStream@<M@>>; }
Stdin = ? FbleStdin <<@>@ M@>(Native@<M@>, Monad@<M@>) { M@<IStream@<M@>>; };

You can't just use a type or type synonym. We need to know the difference
between taking an argument and returning a function.

static FbleValue* FbleStdin(
    FbleValueHeap* heap, FbleProfileThread* profile,
    FbleFunction* function, FbleValue** args);

The problem's that show up right away:
* FbleStdin needs to be a 3 argument function, because it assumes a type for
  M@ that we aren't allowed to assume on the fble side. Or rather, that I
  don't want to be allowed to assume on the fble side.
* FbleStdin wants to return a function with a static value, but if we can do
  that, we either need to let native code call into fble code, or we're back
  to square 1 with defining our own FbleValue* functions which require manual
  block ids.

Oh well. It was worth a try.

We don't need to assume any type for M@. Just use the passed Monad instance
and call return on it. Easy. That's probably a better way to do things anyway.

We don't have to return a function with a static value if we do partial
applies all on the fble side. For example, have 'stdin' return a File@, and

One concern I have with FFI is that people could abuse it. For example, once I
implement some FFI function, what's to prevent anyone from using it and
subverting any type protections?

The answer: have the compiler include the module name in the FFI label. For
example, if my function is 'Foo' and I instantiate the FFI function literal
from module /A/B%, then the FFI function we try to call is /A/B%.Foo. But if
we instantiate an FFI function from some other module /C%, then the FFI
function we try to call is /C%.Foo, which is a completely different function.

For Stdin as above, it would be nice to define a private File@ type that is
constructed and deconstructed entirely on the native side. Now we can do
whatever we want on the native side, so we can use any type at all on the fble
side. For example, Unit@. And we can restrict that Unit@ to the module in
question, so other people can't try to construct wrong things. But that
doesn't prevent us from accidentally constructing the wrong thing.

A suggested workaround: define a single element struct type instead, with some
unique field name to avoid accidentally mistaking two different types.

 @ File@ = *(Unit@ file);

I don't want to define File@ as an Int@ pointer value, because we don't want
people to create them out of thin air.
