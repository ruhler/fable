Standard Library
================
Goal this release: revamp the standard library.

But in what sense?

Completeness:
  Add support for all the kinds of functions people expect in a standard
  library. At least for the data types I already have some support for in my
  library.

Test coverage:
  Full test coverage might be nice?

Documentation:
  Consistent, full, fbld based documentation of all the types and functions in
  the standard library. Maybe also with tutorials or similar style
  documentation.

Goodness:
  Express the library in a good way. In the sense that it composes well and
  generalizes well, so you can reuse it easily for your task.

Performance:
  Consider optimizing the implementation, or understanding any implications of
  API decisions on the performance.

How do I know what's good?
* Compare with other standard libraries for completeness.
* Maybe review changes to other standard libraries over time to see the kinds
  of issues to watch out for?
* Test coverage and documentation should be easy.
* Implementing rosetta code tasks might be a good way to evaluate the library
  once I have a decent first draft of it.

See what the experience is working on those before moving on to the harder

What is a standard library?
* Packages shipped with the language that you can assume will be installed
  anywhere the language is installed?
* Packages owned by the language developers?
* Packages that have matured enough to be generally useful and stable,
  regardless of who develops them?

There is a clear cost to writing code that depends on something other than the
standard library. Users have to download what it depends on. It feels less
trustworthy. So there is a clear benefit of having a comprehensive standard
library people can rely on.

When I was originally organizing code into packages, I was thinking in terms
of minimizing unnecessary dependencies. 'Core' was the minimal set of modules
needed to have a self testing module. Except then I wanted to add more things
like Map and other utilities not needed for testing.

Of the existing packages I have, the following could be reasonable to include
in a standard library:
* Core
* App
* Md5
* Network

The rest are fairly obviously overly specific and not generally useful to
include in a standard library. Though potentially they could be if they
matured and were rewritten with library use in mind.

Should the standard library be a single package, or should we allow it to
include multiple packages? How should the packages of the standard library be
named, and what package path should they use?

For example:
* /Core/Unit%, or /StdLib/Unit%, or /StdLib/Core/Unit%?

I'm tempted to say /StdLib/Unit%, and all of core goes directly into /StdLib,
but that app is a separate package using /StdLib/App%? Except, that would give
App access to std-lib private stuff, which I'd prefer not to have.

So I don't have a good answer for this right now. Which suggests I keep things
as they were.

Another question: should tests of the standard library be included in the
standard library? I don't expect anyone to want to reuse those tests. Seems
like it makes sense to separate those out into another package to help
minimize dependencies?

I like the idea of reserving a part of the module path to designate 'trusted,
stable, standard' library, separate from packages.

* /Std/Core/Unit%? /Standard/Core/Unit%?
* /Std/Unit%?
  This seems like the nicest to use of all I've shown so far.

I'm struggling with how to organize the Int module. I think it ties back to
the question of what my goal is here, now, with the standard library.

Clear goals:
* Flesh out obviously missing functions from the library.
* Clean up existing functions if/as needed with a little thought.
* Remove things that probably shouldn't be there anymore.
* Document all the standard functions.
* Be able to meaningfully write rosetta code examples based on the library.

Beyond that is less clear. What is the end vision for this library?
* The library of useful stuff for me written by me?
* An abstract interface that every implementation and distribution of fble
  must support from here on out?
* A collection of de-facto standard types for interoperability between
  packages?
* A collection of everything you might practically need but don't want to
  write yourself to make writing your own code in fble easy?

In particular I'm struggling with:
* How to organize things into packages, the module hierarchy, and whether they
  ought to be pre-installed or not.

Fble doesn't need to have any builtin library, in theory. In practice you
could argue whether things like Stdio count: it needs to be builtin, does it
also need to be tied to a particular implementation of fble? Depends on
whether I standardize a native interface.

There's also the question of versioning. Any time you have multiple packages
which depend on each other, they have separate versioning, things get tricky.

There's also the issue of things like: String depends on Char depends on Int,
but we want /Int/Show% to depend on /String% while still being grouped
together in the module hierarchy with /Int%. If we say each part of the module
hierarchy is owned by a single package, that would require us to have String
and Int belong to the same package.

Imagine you had a package manager that made it really easy to install fble
packages. In that case, I would want the core datatypes to be in a different
package than 'app' or 'network'. Possibly 'stdio' too, to be honest. At a high
level, we might have:

* core - common data types standardized for use between different packages.
  Unit@, Bool@, Int@, List@, Maybe@, Map@, Char@, String@, Monad@,
  Eq@, Show@, Ord@
* test - test infrastructure.
* random - stuff having to do with random number generation.
* network - stuff having to do with networking.
* app - stuff having to do with applications.
* stdio - stuff having to do with standard input/output.
* md5 - md5 hash, or maybe group into a crypto package.

This makes sense to me. It seems logical. In terms of dependencies between
packages:
* Everything depends on core.
* test depends on stdio

In terms of mapping packages to module paths, we could treat 'core' as a
special case: don't have 'core' in the path. Everything else hosted at
/Std/<name>. Thus:
* /Std/Unit%, /Std/Int%, /Std/Test%, /Std/Random%, /Std/App%, ...

That feels like a good way of doing things.

In terms of what is standard / ships with every implementation of fble...
maybe don't worry or care about that now? We can wait until after we have
package management in place?

I'm still left with questions about submodule organization. There are two, I
suppose:
1. How should we organize, say Int, into submodules?
2. What should be the criteria for including something for Int?

For (2), we can think about Bool instead. We have And, Or, Not. We don't have
Xor, Nand, Nor, Xnor. I think we can reasonably categorize functions into ones
that you would expect to have in the standard library, and ones that are rare.

For example, integer division definitely expected. Integer square root,
definitely not. I've never heard of anyone providing that kind of thing. But I
have implemented integer square root, maybe it's useful elsewhere, it seems
natural to have it live in the core library with the rest of Int. Perhaps in a
separate submodule to avoid pulling it in when not needed. Same could go for
Xnor.

Note: I use Sqrt in the pinball package to solve for roots of a quadratic
equation and otherwise do things you might normal use with a floating point
type instead of an integer type.

I think it's reasonable for me to say: I will have all the functions you
reasonably expect to have in the standard library: And, Or, Not. I will
optionally have additional functions I personally have found useful: Sqrt, and
I will put them in separate submodules where makes sense so you don't pay the
cost for them when you don't use them.

Is that enough for me to go on in terms of int module organization?

I've made good progress, but I seem to have reach a point where everything
feels somewhat slightly tangled and caught up on some core things that don't
feel quite clean to me.

Specifically:
* For Map@, I'm thinking about reusing In@ and Out@ or Output@ for hashing
  keys. Which brings up questions about the difference between Out@ and
  Output@. Maybe there's no confusion? We want just Output@?
* Stdio is waiting on cleanup for places where I convert between binary and
  text streams, which going from things like http text to sockets or strings
  to binary sockets.
* I feel unease about Char@ type class functions. Should we have them or not?
  If so, what do they mean?
* Test@ runner wants to know if it works on Out@ or Put@<Output@>.
* Debug trace wants to know if it should assume Ascii@, or have some Ascii@
  helpers. And also questions whether ToNull, ToString, ToDebug should be
  separated out from /Std/Stream%.

Broadly speaking, it all ties up to /Std/Stream% and character sets:
* Should Output@ be separate from Put@, Get@, In@, Out@?
* When to use Out@ versus Put@<Output@> for things that don't need to close
  the stream?
* What utilities to provide for interpreting characters semantically (e.g.
  IsAlpha) given our vague specification of character set?

Let me remind myself:
* Output@ is for non-interactive stuff. You output all at once.
 - This is appropriate for Show@, Hash@, Serialize@.
* Put@<Output@> is for interactive output that you don't close.
* Out@ is for interactive output that you might close.
* Applications should be explicit about what character types they are using,
  and do their own conversion between characters and binary, because there
  aren't well established standards across existing application domains that
  we could standardize and reuse everywhere.
* Debug trace shouldn't assume Ascii. Better to be a little bit verbose I
  think. Debug trace should be using Output@.
