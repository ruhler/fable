Stdlib Iter
===========
Do we use iterators at all in practice? It feels like old stuff to me.

Current uses:
* Invaders uses it to iterate through a range of integers.
* Drawing Triangles does the same. And rectangles.
* Similar with Sat.

So basically right now, it's only used for integer ranges of things. Maybe we
want to add part of Int@ library for this instead of having a generic integer
type.

No point having iterators for list, you can use ForEach to iterate over the
list.

How about for things like Maps or Sets?

Honestly I feel like each specific kind of thing can implement its own For
and similar traversal functions. I don't see a real benefit to have a unified
generic interface for iteration.

Iterators are useful for traversing an unmaterialized data structure. If we
have a list, a map, or a set, it's already materialized. Just iterate over
that. It's only things like integer ranges where we generate things on the
fly. Or, eventually, input streams.

I suppose an input stream is an abstract iteration. Maybe we can reuse the
same type and infrastructure for both integer iteration and IO iteration. The
important part in both these cases is we want to be able to look at and act
upon earlier elements in the iteration before bringing into memory any
subsequent elements. Or, also, throwing away earlier elements before looking
at subsequent elements. Otherwise you could say 'Range' function produces a
list of integers, or we read an entire file into a list of bytes first.

The 'For' loop is basically an interface for consuming all elements of
something one element at a time. It gives us the one element at a time part.
What it doesn't give us is the ability to break out of the loop early for some
reason. For lists, things like 'Has' where we want to stop after the first
item, currently I'm just operating directly on the list. But maybe it makes
sense to have a generic interface.

Maybe I want to be able to write a generic 'Has' function that works over any
data structure supporting iteration and that can break out early. What
interface would I use for that?

For loops. I use them pretty often. A ForEach like construct that iterates
over integers in a range. Today I separate the range from the iteration. Not

The use cases I have today:

I could also imagine:

  For each integer from a to b with update f.
  While p(s).

The nicest API for my use cases will be for integer range a to b call some
function. It's not super general though. Not nearly so general as a C for
loop, for example.

Another line of thought is, could we use Stream@<A@> as the argument to a
generic for loop? Then provide a library of Stream@ functions?

The important thing about Stream@ is its laziness and streaminess, in contrast
to list. That's exactly my motivation for using Stream@ for IO.

It wouldn't be difficult to produce a stream.

  (Int@, Int@) { Stream@<Int@>; }
  Range = (Int@ l, Int@ h) {
    Lt(l, h).?(false: Return<Int@>);
    Unit@ _ := Put(l);
    Range(Incr(l, h));
  };

That's a super nice way of iterating over things.

But can I easily enough consume the stream? Maybe your base in the for loop is
exactly the stream type we are going to?

Yes. Stream@<Int@> as an argument to for loops, to replace iter, sounds
perfect. Super easy to construct. Super easy to consume from a for loop
implementation point of view. If we want to have a lazy list type, that's easy
to construct too, so that people can easily consume the stream.

I did think about whether we could use a lazy list type instead of stream
everywhere, but it's not quite so nice:
* As a list, it doesn't allow easy concatenation of two different lists.
* We could make it a tree instead, but that's awkward.
* And we don't get flexibility that's nice to have to define your own stream
  type rather than have to go through a lazy list everywhere.

So, plan is:
* Define a /Std/Stream%.ForEach function.
* Define an /Std/Int/Stream%.Range function
 - Just call Range(0, n) rather than define some UpTo function.
* Migrate everyone off of Iter.
* Use streams to replace /Core/Maybe%.For and DoWhile too.

That should get us done with the majority of the 'other modules' from core
library that I need to get rid of.

Let me draft a change. Test it out for Range. /App/Drawing/Test% would be good
to start.
