Import Expression
=================

See background context in thoughts/fble-0.6/monad_composition.txt.

The goal is to be able to import multiple variables at once from a single
structure, to make things more convenient than they already are.

Anyway, long story short, proposal is:

  [type] name [':' name, [',' ...]] '.=' expr ';' expr

Examples:

Instead of:

   Pure = /Core/Random/Prng/Pure%.Pure(Prng);
   M@ = Pure.M@;
   M = Pure.Monad;
   Rng = Pure.Rng; 

We could write:
  
   M@, M: Monad, Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

Instead of:

   Bool@ = /Std/Bool%.Bool@;
   True = /Std/Bool%.True;
   False = /Std/Bool%.False; 

We could write:

   Bool@, True, False .= /Std/Bool%;

Instead of: 
            
  QR@ qr = /Std/Int/Div%.Div(n, Cols);
  Pos@ pos = @(x: qr.r, y: qr.q);

We could write:
            
  Int@ q, Int@ r .= /Std/Int/Div%.Div(n, Cols);
  Pos@ pos = @(x: r, y: q);

Or:

  Int@ x:q, Int@ y:r .= /Std/Int/Div%.Div(n, Cols);
  Pos@ pos = @(x, y);

Key points:
* We still explicitly name all variables being defined, which is important to
  me.
* The definer has full control over what variables are named, not the struct
  they are coming from. With shorthand for the common(?) case where you want
  the names the same.
* It's not position based, because it's just too hard to keep track of
  positions of things, and too risky when positions change.

The question is: does this new syntax provide enough value to justify the
added complexity? It doesn't let you do anything you can't already do, for
example. But then again, neither does case select statements, and those are a
huge improvement in the language.

Some other related things just to note:

* Non-recursive let workaround:
  State@ s := ApplyTo(s.@(...));
  
  instead of:

  State@ ns = s.@(...);
  s = ns;

* Or, I suppose:
  State@ s .= @(s: s.@(...));

More exploration of syntax:

   Pure = /Core/Random/Prng/Pure%.Pure(Prng);
   M@ = Pure.M@;
   M = Pure.Monad;
   Rng = Pure.Rng; 

Or
  
   M@, M: Monad, Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

   ^ /Core/Random/Prng/Pure%.Pure(Prng).^(M@, M: Monad, Rng);

   (M@, M: Monad, Rng) .= /Core/Random/Prng/Pure%.Pure(Prng);

   M@, M, Rng .= /Core/Random/Prng/Pure%.Pure(Prng).^(M@, Monad, Rng);

   .M@, M: .Monad, .Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

   M@, M: Monad, Rng := /Core/Random/Prng/Pure%.Pure(Prng);

   ^ /Core/Random/Prng/Pure%.Pure(Prng).^(M@, Monad(M), Rng);

   do, return .= Monad<S@>;

   M@, M(.Monad), Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

   M@, M(Monad), Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

Syntax wise, this still feels best to me:

   M@, M: Monad, Rng .= /Core/Random/Prng/Pure%.Pure(Prng);

It should be fine to punt this to a later release because:
* It's not backwards incompatible to add later, so no reason to take advantage
  of this clearly backwards incompatible release to do it.
* It doesn't make things easier enough to change how I would design the
  standard library.

I think the syntax I have is the best I can hope for with my requirement of
explicitly naming everything we import.

I think it's worth trying at some point, in case it makes a significant
difference. It doesn't really unlock anything new. We can always desugar to
something like:

   _ = /Core/Random/Prng/Pure%.Pure(Prng);
   M@ = _.M@;
   M = _.Monad;
   Rng = _.Rng;

If I want to justify it from a "fundamentals" point of view, I suppose closest
we could come is to say this is the best we can do to provide abstractions for
importing things. You write a function to return the struct with what you
want, let the user import what they need from that under the names they want.

Experience report:
* I think import highlight dependencies between modules than you would have
  otherwise.
* Having migrated all of the standard library over to import, I certainly
  don't mind having import expression. It's much more compact. Keeps the
  imports small and out of the way. Feels like it makes renaming on import
  more obvious and explicit than it was before. Does especially well when
  importing a lot from the same module, like for lists.
* Renaming modules is much more convenient with import statements, because
  there are a lot fewer places where we name the module that need to be
  updated.
