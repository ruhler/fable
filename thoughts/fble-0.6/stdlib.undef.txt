Undefined
=========
I want some standard library way of saying the equivalent of 'error'.

It causes the program to terminate, ideally with a stack trace. And ideally
you have a decent way in the code to explain the error.

For example:

  Error<Foo@>|'divide by zero';

Here's my current approach:

% DivideByZero = /Core/Undef%.Undef<QR@>;

...
   DivideByZero.undefined

Let's assume we panic, and don't seriously consider at this point the idea of
adding exceptions to fble. What we want is to specify the type, say something
is undefined, and ideally give a explanation of some sort in the code of why
it is undefined, all in some compact way.

If we do an undefined access at the level of code, that could be useful for
debug without a stack trace if you have the location at the point of the
error. If you have a full stack trace, you could call out to another function
that does the undefined access.

How about this though: /Core/Undef%.Undef<X@> returns an undefined polymorphic
function from any type to X@. Then we could use it something like:

   0: Undef<Int@>(Str|'DivideByZero);

It looks just like a function call, but the undefined location is the location
of the function application. We can give any argument we want. So, if you are
okay with string literals, give it one. If not, you could do something like:

  Undef<Int@>(@(DivideByZero: Unit));

Seems ... annoyingly round about to give an indication of the problem though.

I lean towards 'undefined'. Here's how to read it:

  # The undefined value.
  Undefined@ Undefined;

  # Define a named undefined value. It's value really is undefined from the
  # language standard point of view
  Undefined@ DivideByZero;

  # Perform undefined access. This will abort the program.
  DivideByZero.undefined;

The only confusion about the naming conceptually is that DivideByZero is
an undefined value, but DivideByZero.undefined is an undefined behavior.

Maybe Undefined.abort? DivideByZero.abort? DivideByZero.undefined?

How do I feel about 'undef' versus 'undefined'?

% DivideByZero = /Std/Undef%.Undef;
% DivideByZero = /Std/Undef%.Undefined;
% DivideByZero = /Std/Undefined%.Undefined;

Undef@ DivideByZero;
Undefined@ DivideByZero;
Undef@<Int@> DivideByZero;

DivideByZero.undef;
DivideByZero.undefined;

There is precedent for shortening names in the library: Int, Char, Bool. They
all do it. Undef seems reasonable to me. But use it for the field name as
well.

Okay? Cool. Sounds like a plan, and we have three clear use cases:

1. Direct use of undefined:
  /Std/Undef%.Undef<Int@>.undef;

2. Reuse of Undef value:
  % DivideByZero = /Std/Undef%.Undef;
  DivideByZero<Int@>.undef;

3. Reuse of Undef@ type:
  Undef@ DivideByZero;
  /Std/Undef%.Undef@ DivideByZero;

One more question: should the return type go with the field or with the error
itself?

  DivideByZero<Int@>.undef;
  DivideByZero.undef<Int@>;

The first one allows us to encapulate that type if we want. E.g.

  % DivideByZero = /Std/Undef%.Undef<Int@>;
  DivideByZero.undef;

The second feels a little more natural to express things, but a bit more
sketch too. Hmm... I kind of lean towards the second. The undef field is a
poly <@ A@> { A@; } rather than undefined having a type. Makes more sense for
how we expect things to be used.
