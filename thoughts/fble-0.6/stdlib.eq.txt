Equality
========

Today I have Eq@, which is a binary predicate on two items of the same type.
The suggestion is that we are looking specifically for an equality relation,
but in general it could be used for other things like inequalities.

The proposal is pretty simple:

  /Std/Eq%:
   Eq@

And we ask that most data types provide implementations called, Eq in a
/.../Eq% submodule.

Is there anything else worth discussing here?

Use cases:
* Test infra for assertions.
* List methods StartsWith, EndsWith, etc...
* Naturally implemented for Int@ type.
* Presumably others.

The type for Eq@ more generally describes a relation. I don't need to imply
any particular properties, but maybe it's useful to. For example, maybe we
want to use the convention that Eq@ is used for equivalence relations, as
opposed to more generic partial orderings of things.

Does it make sense to define things like:
 <@>@ P@ = <@ T@> { (T@) { Bool@; }; };
 <@>@ R@ = <@ T@> { (T@, T@) { Bool@; }; };

And then use R@ everywhere?

Well, except, R@ isn't nearly as descriptive as Eq@. I prefer Eq@ for my use
cases. We could always define R@ later if it would be useful.

Seems simple enough to me.

If you wanted to be more explicit, we could do something like:

<@>@ Eq@ = <@ T@> {
  *((T@, T@) { Bool@; } eq, (T@, T@) { Bool@; } ne);
};

Now we are saying you need to support both == and !=, and we are giving them
very specific names.

Why would you need to provide both eq and ne when you could always derive one
from the other? I don't know. You might say one is more efficient to implement
than the other, but I don't buy that. The implementations can be identical but
for swapping True with False in everything that is returned?

This seems like unnecessary indirection to me. Why ask everyone to write
eq.eq(...) instead of directly using eq(...)?

Mathematically equivalence is reflexive, symmetric, transitive. That doesn't
tell me anything about what I should call it or what properties it should
support.

---

Discussion on stdlib.ord.txt suggests maybe we want to define:

<@ T@>(Eq@<T@>, T@, T@) Eq = <@ T@>(Eq@<T@> eq, T@ a, T@ b) { eq(a, b); };
<@ T@>(Eq@<T@>, T@, T@) Ne = <@ T@>(Eq@<T@> eq, T@ a, T@ b) { Not(eq(a, b)); };

---

Discussion on stdlib.eq.txt suggests we want:

 /Std/Equality%:
   Equality@(.eq, .ne)
   Equality(function from 'eq' to Equality@?)

To get there is going to be tedious because of how much we define and use Eq@
everywhere. What's the best way to migrate things?

Let's think about how to define a random Equality instance.

For example:

  Eq@<Block@> Block = (Block@ a, Block@ b) {
    /Core/String/Eq%.Eq(a.name, b.name).?(false: False);
    /Core/String/Eq%.Eq(a.file, b.file).?(false: False);
    /Std/Int/Eq%.Eq(a.line, b.line).?(false: False);
    /Std/Int/Eq%.Eq(a.column, b.column).?(false: False);
    True;
  };

Becomes:

  Equality@<Block@> Block = {
    Block@ a, Block@ b <- Equality;
    /Core/String/Equality%.Equality.eq(a.name, b.name).?(false: False);
    /Core/String/Equality%.Equality.eq(a.file, b.file).?(false: False);
    /Std/Int/Equality%.Equality.eq(a.line, b.line).?(false: False);
    /Std/Int/Equality%.Equality.eq(a.column, b.column).?(false: False);
    True;
  };

It's not too bad. But do I really want to use 'Equality' everywhere instead of
'Eq'? Surely /Std/Int/Eq%.Eq.eq ...

Well, I want to distinguish between the equality instance and the equals
function.

  /Std/Int/Eq%.Eq(...) would still work.

It would be a pain to have to define Eq the function and Eq the instance
everwhere, no?

  /Core/String/Eq%.Eq as a function
  /Core/String/Eq%.Equals as the instance? Equal@, Equals@, Equality@ as the
interface?

Honestly, it doesn't feel worth it to me to distinguish between the equals
function and Eq@ interface. If the only difference is a not equals function,
which we can always really trivially get from the equals function, why bother?

Ord@ is a different case. There there are lots of different core functions we
could be interested in: Lt, Gt, Eq, Ne, Le, Ge, Min, Max, Cmp. It's much more
a class of operations. Equality is just equality. So just use equality for it.
That's part of the naming challenge too.
