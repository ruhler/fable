Stdlib Test
===========
How to clean up the unit test API for use in standard library?

The big things are:
* Define test cases.
* Define test suites.
* Implement test runners.

How do I feel about the current approach for defining tests?
* TestResult@ as pass, fail, or xfail)
* TestCase@ as name and function to compute a test result.
* TestSuite@ as name and list of tests.
* Test@ as either a test case or a test suite.

Pass or fail makes sense. Organizing into suites makes sense. The funny bits
are:
1. Having to assign a test a name and look it up by that, rather than using,
say, the name of the test function.
2. Having to pass a function from (Unit@) rather than, say, a function to call
a test?

Let's compare to some existing test frameworks. Java and python are probably
good candidates.

junit features:
* Annotate a method with @Test to mark it as a test case.
  - Reuses the name of the method as the test name.
  - The method takes no arguments.
* Other annotations supported:
  - @ParameterizedTest, @RepeatedTest, @TestFactory, etc.
  - All these things seem like things that normal functions could describe
    just find for us.
* Ability to annotate test as fast or slow.
* A test suite is a class that contains test methods.
* Option to give display names associated with tests.
* Assertions:
  - There are a bunch.
  - assertEquals, fail, assertZero, assertThrows, assertDoesNotThrow,
    assertTrue, assertFalse
  - Including assertions for things like timeout.
* Assumptions.
  - Mark a test as aborted instead of failed.
* Tests can be disabled or conditionally executed.
* Tests run in a particular order (not the order listed).
* And on and on.

The junit approach seems excessively complicated. I have language features to
support these kind of things. No need for special magic annotations and such.

How do I feel about giving an explicit test name and Unit@ function for the
test method?

It hasn't been a big problem for me so far. But it would be much nicer if vim
could jump write to the test that failed. If we could somehow magically get
the file and line number of the test case, that would be amazing for practical
use.

I can imagine a world where we need language support for dealing with
untrusted code. Untrusted code could crash, it could take too much wall time,
too much cpu time, too much memory. Code under test is untested code. An OS
will run app code, which is untrusted code. If we have some language feature
to deal with untrusted code, that would be useful here.

To be really useful, we would want some way to get access to the stack trace
associated with the failing code. Maybe there is a timeout, cpu, or memory
quota that results in a stack trace. Then we can run code and get the stack
trace in case of failure. Then we could report that stack trace as part of the
test failure.

But I don't have such a feature today. Should I add one? What would it look
like? Are there performance implications for supporting such a feature? How
about obfuscation implications? Like, doesn't that give too much information
about the untrusted code? You could enumerate the whole call hierarchy if you
wanted.

Or, we could say you get the call stack only if there is debug symbols. But
that doesn't help for running a test suite with no debug symbols. Could we
rely on that? Or maybe, for tests that fail without debug symbols, we just say
the test crashed, and if you want to debug it, run it with debugging?

In practice, having stack traces without having to rerun something under a
debugger is hugely valuable.

I have an Output@ type now. I could imagine a set of functions:

  <@>@ Result@ = <@ T@> { +(T@ result, Output@ error); };
  <@ A@, @ B@>((A@) { B@; }, A@) { Result@<B@>; } Try;
  (Int@)<@ A@, @ B@>((A@) { B@; }, A@) { B@; } Timeout;

Where Timeout takes a wall clock time, or a cpu time, or a memory use, it runs
the function, but crashes if the function doesn't finish in time.

These don't feel like deterministic functions though, do they? Should they be?
Should it be done in some IO monad?

Crashes should be deterministic. Resource usage not, but it could be if we do
it on instruction counts and allocation counts maybe. As long as it isn't
influenced by GC. Maybe timeout can be used with whatever deterministic
profiler we have?

Anyway, 'Try' function is all I would need to start for a test framework. Run
each test under Try, catch the error message, go from there.

But the error message is going to be a crash, not a user provided error
message. Which means its use is a generalization, not a replacement, for my
proposed approach. We shouldn't be relying on this for test.

All of which leads me to:
* For now, let's just stick with the current testing approach.
* In the future, if need be, we can add support for a 'Try' function to make
  testing a little more useful.

Other questions for defining tests:
* Should tests run in IO monad? No. I think definitely not. Generalize the
  things you want to test so we can run them purely. It's just that super big
  input test case where that doesn't work well, because we would need big pure
  data.

In summary, for defining tests:
* Stick with what we have today.
* We don't get nice error: messages that bring you right to the test case. Oh
  well.
* Buggy code under test brings down the entire test infra. Oh well.
* You have to remember to put code under test behind a function. Oh well.

It's too bad it doesn't compose well. Like, you can't compose buggy code with
working code to get working code. As soon as something is buggy, you lose it
all.

Could we compromise? How about a 'Try' function that takes a default value and
returns that? We could run all our tests under:

  Try(Fail(Str|'crashed'), test.run, Unit);

No need to worry about capturing stack traces. The user could go back and run
it under a debugger... Ugh.

Maybe put Try under a /Debug% module, have it output the stack trace in debug
when it runs?

I suppose today my unwinding implementation prints the crash to stderr as it
unwinds. It doesn't collect it anywhere. Presumably that would need to change
if we wanted something like 'Try' to be implemented for legitimate use cases.

Let's review.

* I have an API today where test names are explicit, bodies are behind a
  (Unit@) { ... } function, and if a test crashes, the whole test runner
  crashes.
* It might be nice if test name was inferred. But that's less valuable in case
  of anonymous tests. Like, no benefit of TestFoo = (...) { ... } over
  Test(Str|Foo, (...) { ... }) aside from slight syntactic overhead.
* It would be nice to get an error pointing directly to the test in question.
  Perhaps via a special |^ kind of literal, whose semantics, implementation,
  justification, and generalizability are unclear at the moment.
* It would be nice not to have the syntactic overhead of (Unit@ _ ) { ... }
  everywhere.
* It would be nice if one test crashing doesn't prevent other tests from
  running.
* It would be nice when a test fails for any reason to get a stack trace of
  the test failure, which we can pragmatically capture and report in whatever
  format the test runner thinks is best.

All these things that would be nice are hard and slightly weird. Let's start
thinking outside the box. Because lots of little un-nice things add up to a
not nice experience. I would love to have a nice experience for unit testing.

Idea: how about don't put test bodies behind (Unit@ _). A Test@ becomes a test
result. The test runner doesn't run the test code, it just prints out the
results.
* No need for (Unit@ _ ) {...} everywhere, which is nice.
* We get the same end result as we already do today, practically speaking. The
  list of test results with a crash when one of them crashes.
* But: a crash will prevent us from seeing any test results, not just the
  tests that come after the failure.
* But: a hang will prevent us from seeing any test results, not just the tests
  that come after the hang.
* But: we'll have no idea from a hang which test failed.
* But: if tests are slow to all run, we get no indication of progress.

Having a special test runner makes sense. We do the same kind of thing for
Stdio@, App@, etc: You give a value of specific type (Test@ in this case), the
test runner is a C program that knows how to interpret it. The C program will
be capable of catching exceptions: FbleApply returns NULL in that case.

As a happy side effect of printing stack traces to stderr on test failure,
we'll have them as part of the test output. We just can't manipulate them
programmatically.

So, benefits of this approach:
* It makes sense conceptually.
* It let's us catch crashes in test runs.
* It let's us see stack traces.

Downsides:
* We don't have control over how stack traces are printed.
* We could also write a pure test runner (like today). Which approach should
  we invest in? Because it doesn't seem like we can easily share.

To do everything in fble, we need access to stack traces in fble. But what is
a stack trace even? If you have obfuscation, or your strip symbols. We want to
be able to completely strip symbols to avoid excess code size. We could report
instruction pointers, make whoever has the symbols recover the stack trace
after, but then we can't print something pretty to the user.

The only way we can write an in-fble test runner that can print pretty stack
traces to the user is to specify stack traces as part of the language
definition somewhere and require it to be supported.

Stepping back again, in the hopes it can provide insight as to how to move
forward here. The main challenges with the test framework at the moment:
* Tediousness of (Unit@ _) { ... } syntax.
  - Could be solved with ${ ... }, but that seems maybe hacky.
* If a test crashes, it prevents other tests from running.
  - Could be solved from C with custom test runner. Feels clunky.
  - Could be solved with 'try' feature to language.
  - Could be solved with builtin try function?
  - But note: we never solve the issue of test infra itself crashing.
  - Maybe there's a way to dynamically load test lists to catch issues of test
    list generation crashing.
* Programmatic access to debug info
  - Requires we make debugging part of the fble spec somehow.
* Assuming I implement the test runner in fble, I need to sort out the
  standard apis for stream outputs, which is more work to do (I need to do it
  eventually).

Some other things to keep in mind:
* The user has to annotate which code is part of running which test somehow.
  There's no getting around that. If there's a crash, we need to assign it to
  a particular test, or the test infra. Use of (Unit@ _) { ... } is the best
  way to do that in the API.
* Non-programmatic access to debug info, like seeing it as stderr from the
  test implementation, is not useful in general. I already have that as a
  hacky solution.
* Explicit embedding of __FILE__ and __LINE__, and maybe __MODULE__ could be
  very convenient for producing error messages. It is not a complete solution
  to programmatic access to debug info.
* Explicitly named tests makes perfect sense. There is no advantage to
  inferring the name from variable declarations. Except maybe a slight
  syntactic one that you should be able to use literals to solve, otherwise
  literals aren't doing their jobs.
